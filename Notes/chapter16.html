Chapter 16

Logic Programming

Declarative programming through symbolic logic. Programs describe what is true, not how to compute it. Prolog uses Horn clauses, unification, and backward chaining to answer queries.

Predicate Calculus Horn Clauses Prolog Unification Backtracking List Processing Declarative
🔑 Central Distinction

Logic programming is declarative, not procedural. You specify the form of results, not the steps to produce them. The language's inference engine figures out how.

Predicate Calculus

Proposition
A logical statement that may or may not be true. Consists of one or more objects and the relationships among them.

First-order predicate calculus is the form of symbolic logic used for logic programming. It serves three needs: (1) express propositions, (2) express relationships between propositions, (3) describe how new propositions can be inferred from others.

Atomic Propositions

An atomic proposition has two parts: a functor (the relationship name) and an ordered list of parameters.

% Atomic proposition examples (Prolog syntax)
student(jon)             % "jon is a student"
like(seth, osx)          % "seth likes osx"
like(nick, windows)
like(jim, linux)

Forms of a Proposition

Fact
  • The proposition is assumed to be true
  • Example: student(jon) — "Jon is a student (unconditionally)"
Query
  • The truth of the proposition is to be determined
  • Example: "Can you prove that Jon is a student?"

Horn Clauses

Horn Clause
A restricted form of proposition with at most one positive literal. Used because they can represent most useful propositions and support efficient inference.
Headed Horn Clause (Rule)
  • One consequent (left/head) and a conjunction of antecedents (right)
  • States relationships / derives new facts
  • Prolog syntax: head :- body
Headless Horn Clause (Fact)
  • No head — just an unconditional fact
  • States something that is always true
  • Prolog syntax: fact(args).
% Rule (headed Horn clause)
likes(bob, trout) :- likes(bob, fish), fish(trout).
% "Bob likes trout IF bob likes fish AND trout is a fish"

% Fact (headless Horn clause)
father(bob, jake).
% "Bob is Jake's father" (unconditionally true)

Prolog Overview

Prolog development started in the early 1970s at two institutions simultaneously:

  • University of Aix-Marseille: Colmerauer & Roussel — focused on natural language processing
  • University of Edinburgh: Robert Kowalski — focused on automated theorem proving

We use Edinburgh syntax. Two main dialects exist; Edinburgh is the most common.

🔑 Declarative Semantics

Each Prolog statement has a simple, context-independent meaning. This is why logic programming semantics is simpler than imperative language semantics — a statement means the same thing regardless of where it appears in the program.

⚠ Variable Instantiation

Prolog variables are not like imperative language variables. Instantiation — binding a variable to a value — only happens during the resolution process. A variable can be instantiated to a specific value, then uninstantiated again on backtracking.

Fact Statements

Headless Horn clauses — unconditional truths that form the knowledge base.

female(shelley).
female(anna).
male(bill).
father(bill, jake).
father(bill, shelley).

Rule Statements

Headed Horn clauses. Head (consequent) on left; body (antecedent) on right. Commas in body mean AND.

% Specific rule (atoms only)
ancestor(mary, shelley) :- mother(mary, shelley).

% General rules (variables — universal scope)
parent(X, Y) :- mother(X, Y).
parent(X, Y) :- father(X, Y).
grandparent(X, Z) :- parent(X, Y), parent(Y, Z).

% Sibling rule
sibling(X, Y) :- mother(M, X), mother(M, Y),
                 father(F, X), father(F, Y).

Goal Statements

Queries — propositions Prolog must try to prove. The ?- is the query prompt.

?- man(fred).             % simple fact check
?- parent(X, mike).       % find all X such that X is Mike's parent
?- parent(X, mike), male(X).  % conjunctive goal (two subgoals)

Prolog Inferencing Process

Bottom-up (Forward Chaining)
  • Start from facts, derive new facts, until goal is reached
  • Works well with large sets of relevant facts
  • Prolog does NOT use this
Top-down (Backward Chaining) ← Prolog
  • Start from the goal, find rules that derive it, recurse into their conditions
  • Works well with small sets of relevant facts
  • Prolog uses this approach

Subgoal Search Strategy

Breadth-first
  • Work on all subgoals in parallel
  • Uses more memory
  • Prolog does NOT use this
Depth-first ← Prolog
  • Prove first subgoal completely, then second, etc.
  • Fewer computational resources
  • Prolog uses this approach

Unification & Backtracking

Unification
The process of finding values for variables in propositions that allow a matching process to succeed. A variable is instantiated to a tentative value. If matching fails, backtracking occurs and a different value is tried.

Theorem Proving

Resolution is not only a query mechanism; it is also a formal theorem-proving technique.

Refutation Complete
If a contradiction exists in the propositions under consideration, resolution can detect it. That is why it supports proof by contradiction.
  1. Start with the known hypotheses.
  2. Negate the theorem you want to prove.
  3. Add that negated theorem to the hypotheses.
  4. Use resolution to derive a contradiction.
Proof Pattern

If assuming the theorem is false creates an inconsistency, the original theorem must be true.

Tracing Model

Prolog has a built-in trace mechanism with 4 events:

EventMeaning
CallBeginning of an attempt to satisfy a goal
ExitGoal has been successfully satisfied
RedoBacktracking — revisiting a previous goal
FailGoal cannot be satisfied
 
likes(jake, chocolate).
likes(jake, apricots).
likes(darcie, licorice).
likes(darcie, apricots).

?- likes(jake, X), likes(darcie, X).

% Execution trace:
(1) Call:  likes(jake, _0)?          % try to satisfy first subgoal
(1) Exit:  likes(jake, chocolate)    % X = chocolate
(2) Call:  likes(darcie, chocolate)? % try second subgoal
(2) Fail:  likes(darcie, chocolate)  % darcie doesn't like chocolate
    Redo:  likes(jake, _0)?          % backtrack — try next value for X
(1) Exit:  likes(jake, apricots)     % X = apricots
(3) Call:  likes(darcie, apricots)?
(3) Exit:  likes(darcie, apricots)   % success!

X = apricots
📝 Subgoal Order Matters

Placing more restrictive subgoals first reduces unnecessary backtracking and improves efficiency. The order of facts and rules in the file also affects which solution is found first.

Simple Arithmetic

The is operator: evaluates an arithmetic expression on the right side and unifies the result with the variable on the left. 

A is B / 17 + C.   % B and C must already be instantiated; A must not be
⚠ is ≠ Assignment

is is not assignment. You cannot write Sum is Sum + Number. because Sum is already instantiated on the left. Prolog variables are bound once per resolution path, not reassignable. 

speed(ford, 100).   speed(chevy, 105).
time(ford, 20).     time(chevy, 21).

distance(X, Y) :- speed(X, Speed),
                  time(X, Time),
                  Y is Speed * Time.

?- distance(chevy, X).
% Speed = 105, Time = 21 → X = 2205

List Structures

Lists are the second fundamental data structure in Prolog. A list is a sequence of any elements — atoms, structures, or other lists.

NotationMeaning
[apple, prune, grape]Simple list of atoms
[]Empty list
[X]List with one element X
[X | Y]List with head X and tail Y (Y is a list)
[X1, X2 | Y]First two elements X1 and X2, remainder is Y
📝 Writing List Predicates — Rules of Thumb
  • No "returns" — the result is always a parameter
  • Base case and recursive case are separate statements
  • Put the base case first — Prolog matches top to bottom
  • Use _ (anonymous variable) when you don't care about a value

Strategy: X is a member if it's the head, OR if it's a member of the tail.

member(Element, [Element | _]).         % base: element IS the head
member(Element, [_ | Lis]) :-           % recursive: check the tail
    member(Element, Lis).

% Usage
?- member(a, [b, a, c]).   % true
?- member(X, [a, b, c]).   % X = a ; X = b ; X = c

Strategy: Appending an empty list yields the second list unchanged. Otherwise, prepend the head of the first list to the result of appending the tail.

append([], Lis, Lis).                             % base case
append([Head | Lis_1], Lis_2, [Head | Lis_3]) :-  % recursive case
    append(Lis_1, Lis_2, Lis_3).

% Works in multiple directions!
?- append([a, b], [c, d], X).           % X = [a,b,c,d]
?- append([a, b], X, [a, b, c, d]).     % X = [c,d]
?- append(X, [c, d], [a, b, c, d]).     % X = [a,b]

Strategy: Reverse the tail, then append the head to the end of that reversed tail.

reverse([], []).                             % base case
reverse([Head | Tail], List) :-              % recursive case
    reverse(Tail, Result),
    append(Result, [Head], List).

?- reverse([a, b, c], X).   % X = [c,b,a]
?- reverse(X, [a, b, c]).   % X = [c,b,a]

Deficiencies of Prolog

Prolog always matches facts/rules top-to-bottom and subgoals left-to-right. The programmer can influence efficiency by ordering rules (more likely to succeed = higher) and subgoals (more restrictive = earlier). But there's no explicit control flow — you can't say "try rule B before rule A."

Prolog can only prove things using known facts and rules. If it answers false, that means it cannot prove it — not that it is actually false.

⚠ False ≠ Actually False

A query returns "false" if the required facts are simply absent from the database — even if the statement would be true in the real world. This is a fundamental epistemological limitation.

Consider this sibling rule that produces wrong results:

sibling(X, Y) :- parent(M, X), parent(M, Y).
?- sibling(X, Y).
X = jake, Y = jake.    % jake is his own sibling?!

Fix attempt using not:

sibling(X, Y) :- parent(M, X), parent(M, Y), not(X=Y).
⚠ not ≠ Logical Negation

Prolog's not succeeds when its argument fails to be proven. not(X=Y) means "there is no fact proving X equals Y" — it does NOT mean X and Y are truly unequal. Establishing true inequality for all atom pairs would require an impractical number of facts.

Some tasks can be specified in Prolog but are horribly inefficient. The sorting example:

sort(Old, New) :- permute(Old, New), sorted(New).
sorted([]).
sorted([X]).
sorted([X, Y | List]) :- X =< Y, sorted([Y | List]).

This "sorts" by trying all permutations until one is sorted — effectively brute-force, O(n!) work. A real sort algorithm requires procedural control that Prolog makes awkward to express.

Applications of Logic Programming

Logic programming is especially strong in domains where stored relationships and inference matter more than low-level execution speed.

Relational Databases
  • Facts model stored data; rules model relationships between tables
  • Queries are naturally nonprocedural
  • Deductive capabilities go beyond ordinary static storage
Trade-off
  • Inferencing is slower than direct optimized table lookup
  • Expressiveness comes with performance cost
Expert Systems
  • Rules + facts fit the knowledge-base model directly
  • Trace support can explain conclusions
  • New facts and rules can simulate learning
Natural-Language Processing
  • Grammar rules align well with logic rules
  • Backward chaining can assist parsing
  • Clausal forms help model sentence meaning

Chapter 16 Summary

  • Logic programming is declarative: Specify what is true, not how to compute it. The inference engine handles the how.
  • Horn clauses: Facts (headless) and rules (headed). Facts are unconditional truths; rules are conditional derivations. Prolog uses Horn clause syntax.
  • Prolog basics: Atoms start lowercase; variables start uppercase. Three statement types: facts, rules, goals.
  • Resolution: Backward chaining (goal → facts) + depth-first search + backtracking when subgoals fail.
  • Theorem proving: Negate the target theorem, add it to the hypotheses, and derive a contradiction through resolution.
  • Unification: Matching variables to values. Instantiation is temporary — backtracking un-does it.
  • List processing: Uses [Head | Tail] pattern. member, append, reverse follow a base-case + recursive-case structure.
  • Key deficiencies: Closed-world assumption (false ≠ actually false), negation as failure (not logical NOT), and brute-force inefficiency for some tasks.
  • Best-fit applications: deductive databases, expert systems, and natural-language processing.
