Chapter 7

Expressions & Assignment Statements

Expressions compute values. Assignment stores them. But the design decisions around evaluation order, type conversion, operator overloading, side effects, and referential transparency have far-reaching consequences.

Arithmetic Expressions

An arithmetic expression consists of operators, operands, parentheses, and function calls. The design decisions are about how to evaluate them.

Operators can be classified by their position relative to their operands:

NotationPositionExample
PrefixOperator before operands+ a b (LISP: (+ a b))
InfixOperator between operandsa + b (most languages)
PostfixOperator after operandsa b + (Forth, RPN calculators)

Operator Precedence and Associativity

When an expression contains more than one operator, rules determine which operations are performed first.

Operator Precedence
The hierarchy of operators that determines the order in which operators of different precedence levels are evaluated. Higher-precedence operators bind more tightly to their operands. Example: a + b * c evaluates as a + (b * c) because multiplication has higher precedence than addition.
Operator Associativity
When two operators of the same precedence level appear in sequence, associativity determines evaluation order. Left-associative: a - b - c = (a - b) - c. Right-associative: a ** b ** c = a ** (b ** c) (exponentiation is typically right-associative).
Ruby & APL exceptions

In Ruby, all operators have equal precedence (all are actually method calls). APL evaluates all operators right to left, regardless of precedence. These are intentional design choices with significant implications for readability.

Unary Operators

Unary minus (-x) can be ambiguous in expressions like -a ** 2. Should this be (-a) ** 2 or -(a ** 2)? Most languages make unary minus lower precedence than exponentiation, giving -(a ** 2).

Operand Evaluation Order

When evaluating an expression like f(a) + g(b), should f(a) or g(b) be evaluated first? Most languages do not specify operand evaluation order, leaving it to the compiler. This is fine unless operand evaluation has side effects.

⚠ Side Effects and Operand Order

If f(a) modifies a variable that g(b) uses, the result of the expression depends on evaluation order — and that order is compiler-defined, not language-defined in most languages. This is a source of subtle, non-portable bugs in C and C++.

/* C: undefined order of evaluation */
int a = 5;
int result = (a = a + 1) + (a * 2);  /* Undefined! */

Overloaded Operators

Operator Overloading
Using a single operator symbol for more than one purpose, with the meaning determined by the types of the operands. Example: + means integer addition for int operands, floating-point addition for float operands, and concatenation for strings in some languages.

Most languages have some built-in operator overloading. The design question is whether user-defined overloading is allowed:

  • C++, Python, Ruby: allow user-defined operator overloading — programmers can define + for custom types.
  • Java: does NOT allow user-defined operator overloading (but uses it internally for + on strings). Rationale: user overloading reduces readability when operators behave unexpectedly.
  • Ada: allows overloading of predefined operators for user-defined types.
Advantage of operator overloading
  • Natural, readable notation for custom types (e.g., matrix arithmetic)
  • Allows user types to behave like built-in types
Disadvantage of operator overloading
  • Reduces readability if operator meaning is not obvious from context
  • Can obscure performance differences (a + that does O(n) work)

Type Conversions

When values of different types appear in expressions, the language must decide how to handle the type mismatch.

Narrowing Conversion
Converting a value to a type that cannot store all values of the original type. Example: float to int loses the fractional part; int to short may lose magnitude. Potentially unsafe — should always require an explicit cast.
Widening Conversion
Converting a value to a type that can store all values of the original type. Example: int to float, int to double. Safe — no information is lost (though precision may be affected for very large integers). Can be done implicitly.
Coercion (implicit conversion)
An implicit type conversion performed automatically by the language. Example: in 3 + 4.0, the 3 is coerced (implicitly converted) to 3.0 so that floating-point addition can be performed. Coercion is widening in this case, which is safe. Languages vary greatly in how much coercion they allow — more coercion means more writability but potentially less reliability (type errors may go undetected).
Explicit Type Casting
A programmer-specified type conversion. In C: (int) 3.7 → 3. In Java: (int) 3.7. Explicit casts are required for narrowing conversions and signal to the reader that a deliberate type change is happening.

Relational and Boolean Expressions

Relational Expressions

A relational expression has two operands and a relational operator (<, >, ==, !=, etc.) that evaluates to a Boolean result. Design issue: in C, there is no Boolean type — relational expressions return 0 or 1 (integers). This allows nonsense like a < b < c which compiles but does not test whether b is between a and c.

Boolean Expressions

Combine relational or Boolean values with AND, OR, NOT. In C, any non-zero integer is true; in Java/Python, Booleans are a distinct type.

Short-Circuit Evaluation

Short-Circuit Evaluation
The result of a Boolean expression is determined without evaluating all operands if the result can be established from the operands evaluated so far. A AND B: if A is false, B is not evaluated. A OR B: if A is true, B is not evaluated.

Why it matters

Short-circuit evaluation enables safe idioms:

/* C — short-circuit prevents null pointer dereference */
if (ptr != NULL && ptr->value > 0) { ... }
/* if ptr is NULL, ptr->value is never evaluated */
# Python — same pattern
if lst and lst[0] == target:  ...

Without short-circuit evaluation, the second operand would always be evaluated, causing a runtime error if the pointer is null or the list is empty.

Ada uses and then / or else for short-circuit; plain and / or are not short-circuit. This makes the intent explicit.

Side Effects

⚠ Exam Focus — also appears in Chapter 15

Side effects and referential transparency are important topics that connect imperative and functional programming paradigms. Make sure you understand both concepts and can apply them.

Side Effect
A side effect occurs when the evaluation of an expression (or function call) modifies some state outside its local scope — e.g., changes a global variable, modifies a parameter, performs I/O, writes to a file. The function does something beyond just returning a value.

Examples of side effects

/* C — function with a side effect: modifies global variable */
int count = 0;
int increment() {
    return ++count;  /* side effect: modifies global 'count' */
}

int result = increment() + increment();
/* Result depends on whether left or right operand is evaluated first!
   If count starts at 0:
   - Left first:  (1) + (2) = 3, count = 2
   - Right first: (2) + (1) = 3, count = 2
   Result is same here, but this is coincidental.

   With: increment() + count  →  either 1+1=2 or 1+2=3 depending on order */

Side effects make programs harder to reason about, test, and parallelise. The result of an expression may depend on when it is evaluated, not just what its arguments are.

Referential Transparency

Referential Transparency
A property of an expression: it is referentially transparent if it can be replaced with its value without changing the behaviour of the program. Equivalently: a function is referentially transparent (also called a pure function) if its return value depends only on its input arguments and it has no side effects.

Example — referentially transparent

/* Pure function — no side effects, result depends only on input */
int square(int x) { return x * x; }

square(5) + square(5)  /* can be replaced by 25 + 25 */

Example — NOT referentially transparent

/* Function with a side effect */
int counter = 0;
int nextVal() { return ++counter; }

nextVal() + nextVal()  /* NOT replaceable — each call returns a different value */

Why referential transparency matters

  • Reasoning: you can substitute equals for equals, as in mathematics.
  • Testing: a pure function is fully determined by its inputs — easy to unit test.
  • Parallelism: pure functions can be evaluated in any order or in parallel without affecting correctness.
  • Functional languages: Haskell enforces referential transparency entirely. Scheme encourages it. Purely functional programs have no side effects — all computation is through function evaluation.
Referentially transparent (pure)
  • Predictable, easy to reason about
  • Order of evaluation does not matter
  • Trivially safe to parallelise
  • Easy to cache/memoize results
With side effects (impure)
  • Result depends on program state and evaluation order
  • Hard to parallelise safely
  • Interaction with I/O is inherently a side effect
  • Necessary for practical imperative programs

Assignment Statements

Simple Assignment

In most imperative languages: variable = expression. The expression on the right is evaluated and the result is stored in the variable on the left (the l-value).

l-value vs r-value
An l-value is a value that can appear on the left side of an assignment — it refers to a memory location. An r-value is a value that can appear on the right side — it is the value stored at a location (or a literal). Variables are both l-values and r-values; constants and expressions are only r-values.

Compound Assignment Operators

Shorthand forms that combine an arithmetic operation with assignment: a += b (same as a = a + b). C, C++, Java, Python all support these. They reduce redundancy and improve writability, but add to the number of constructs a reader must know.

Unary Assignment Operators

C-style ++ and --: increment/decrement by 1. The pre- (++a) vs post-increment (a++) distinction affects the value of the expression itself:

int a = 5;
int b = ++a;  /* b = 6, a = 6 (pre-increment: increment first) */
int c = a++;  /* c = 6, a = 7 (post-increment: use then increment) */

Multiple Assignment

Some languages allow assigning the same value to multiple variables:

/* Python — tuple unpacking (multiple assignment) */
a, b = 5, 10
a, b = b, a  /* swap without temporary variable */
/* C/Java — cascaded assignment (right-associative) */
a = b = c = 0;  /* same as a = (b = (c = 0)) */

Assignment as an Expression

In C, C++, and Python (walrus operator :=), assignment is an expression with a value. This enables:

/* C — assignment in loop condition */
while ((ch = getchar()) != EOF) { process(ch); }

This is expressive but a reliability concern: if (a = 5) in C is a legal assignment, not comparison. A common bug. Java and C# prevent this by requiring the condition of an if to be a Boolean type.

Mixed-Mode Assignment

When the type of the expression does not match the type of the target variable, an implicit conversion (coercion) may occur:

/* Java — widening is allowed implicitly */
int i = 5;
double d = i;  /* int → double, safe widening coercion */

/* Java — narrowing requires explicit cast */
double d = 3.7;
int i = (int) d;  /* Explicit cast required; result is 3 (truncation) */

More permissive languages (like early C) allowed almost any assignment with implicit coercion, reducing reliability. Stricter languages (Ada, Haskell) require explicit conversions for virtually all type changes.

Quick Check

A function f(x) always returns the same value for the same input and modifies no external state. This function is best described as:

  • A Referentially transparent (pure)
  • B Polymorphic
  • C Generic
  • D Idempotent

Chapter 7 Summary

  • Operator precedence: determines evaluation order for different-precedence operators.
  • Associativity: determines evaluation order for same-precedence operators (usually left-to-right).
  • Operand evaluation order: usually unspecified; dangerous when operands have side effects.
  • Overloading: same operator symbol, different meanings depending on operand types. User-defined overloading: C++/Python yes; Java no.
  • Widening: safe implicit conversion (int → float). Narrowing: potentially unsafe, should require explicit cast.
  • Coercion: implicit type conversion. More coercion = more writability but less reliability.
  • Short-circuit: Boolean operator stops evaluating when result is determined. Critical for null-safety idioms.
  • Side effect: expression evaluation changes state beyond returning a value. Makes programs order-dependent and harder to reason about.
  • Referential transparency: expression can be replaced by its value. Requires no side effects. Foundation of functional programming.
  • Assignment: l-value (memory location) gets r-value (computed value). Multiple assignment and assignment-as-expression are language-specific features.
