Chapter 12

Support for Object-Oriented Programming

Object-oriented programming organises code around objects that bundle state and behaviour. How different languages support — or constrain — OOP reveals fundamental design trade-offs between purity, flexibility, performance, and safety.

Introduction to OOP

Object-oriented programming is built around three core ideas that, together, provide a powerful approach to managing software complexity:

  1. Encapsulation — bundling data and operations that manipulate that data into a single entity, and restricting access to internal details.
  2. Inheritance — defining new classes in terms of existing ones, inheriting their data and methods.
  3. Polymorphism (dynamic dispatch) — a reference to a parent class can refer to objects of any subclass, and the correct method is selected at runtime.

The combination of these three mechanisms is what makes OOP powerful for large-scale software development.

Core OOP Concepts

Encapsulation and Data Hiding

Encapsulation
Grouping related data (attributes/fields) and operations (methods) into a single unit (class/object), with a public interface that hides implementation details. The internal state is protected from direct external access.

Encapsulation provides two benefits:

  • Information hiding: implementation details can change without affecting code that uses the class (as long as the interface stays the same).
  • Integrity: the class controls all access to its data, which allows it to enforce invariants (e.g., a non-negative balance).

Inheritance

Inheritance
A mechanism by which a new class (subclass, derived class, child class) can be defined as an extension of an existing class (superclass, base class, parent class). The subclass inherits all public and protected members of the superclass, and can add new members or override (redefine) inherited methods.
// Java — inheritance
class Animal {
    String name;
    void speak() { System.out.println("..."); }  // can be overridden
}

class Dog extends Animal {
    void speak() {                    // overrides Animal.speak()
        System.out.println("Woof!");
    }
    void fetch() { ... }              // new method
}

Method Overriding

A subclass provides its own implementation of a method defined in the superclass. The override replaces the inherited implementation for objects of the subclass type.

Not the same as overloading: overloading uses the same name with different parameter types (resolved at compile time); overriding uses the same name AND same parameter types (resolved at runtime via dynamic dispatch).

Polymorphism

Polymorphism
The ability to use a reference of a parent (superclass) type to refer to objects of different subclass types. A single variable name can refer to objects of many different (sub)types, and the appropriate behaviour is determined at runtime.
// Java — polymorphism
Animal a;           // reference of type Animal

a = new Dog();
a.speak();          // calls Dog.speak() → "Woof!"

a = new Cat();      // Cat extends Animal, overrides speak()
a.speak();          // calls Cat.speak() → "Meow!"

This is only possible because of dynamic binding — the method to call is not determined by the declared type of the variable, but by the actual type of the object at runtime.

Dynamic Binding (Late Binding)

Dynamic Binding (Late Binding)
The binding of a method call to the method definition is determined at runtime, based on the actual type of the object receiving the call, rather than the declared type of the variable. Required for polymorphism.
Static Binding (Early Binding)
The binding of a method call to its definition is determined at compile time, based on the declared type of the variable. Non-virtual methods in C++, and all method calls in non-OOP languages, use static binding.

C++ virtual keyword

// C++ — dynamic binding requires 'virtual'
class Animal {
public:
    virtual void speak() { cout << "..."; }   // dynamic binding
    void breathe()       { cout << "hh..."; } // static binding (no virtual)
};

class Dog : public Animal {
public:
    void speak() override { cout << "Woof!"; }
};

Animal* a = new Dog();
a->speak();    // "Woof!" — dynamic binding
a->breathe();  // Animal::breathe() — static binding

In Java and C#, all instance methods are virtual by default (dynamic binding unless the method is declared final/sealed). C++ requires explicit virtual; non-virtual methods in C++ use static binding for performance.

Implementation — vtable

Dynamic binding is typically implemented using a virtual function table (vtable): each class has a table of pointers to its virtual method implementations. An object contains a pointer to its class's vtable. Dispatching a virtual call looks up the correct method pointer at runtime. This adds a small, constant overhead compared to a direct call.

Abstract Classes and Interfaces

Abstract Class
A class that cannot be instantiated — it exists only to be subclassed. It may declare abstract methods: methods with no implementation that subclasses must provide. An abstract class can also have concrete (implemented) methods.
// Java abstract class
abstract class Shape {
    abstract double area();          // must be overridden by subclass
    void describe() {
        System.out.println("Area: " + area());  // concrete method
    }
}

class Circle extends Shape {
    double radius;
    Circle(double r) { radius = r; }
    double area() { return Math.PI * radius * radius; }
}

// Shape s = new Shape();    // ERROR — cannot instantiate abstract class
Shape s = new Circle(5.0);   // OK — polymorphism
s.describe();                // calls Circle.area() via dynamic binding
Interface
A completely abstract type that specifies a set of method signatures that implementing classes must provide. Unlike abstract classes, a class can implement multiple interfaces, allowing a form of multiple inheritance of types without multiple inheritance of implementation.
// Java interface
interface Drawable {
    void draw();           // abstract method (implicit)
    default void hide() {  // default method (Java 8+)
        System.out.println("Hidden");
    }
}

class Circle extends Shape implements Drawable, Serializable {
    public void draw() { System.out.println("Drawing circle"); }
    // ...
}

Design Issues for OOP Languages

Exclusivity of Objects

How completely does the language commit to "everything is an object"?

  • Pure OOP (Smalltalk, Ruby): everything is an object, including integers, characters, classes, and even nil. Operations on all values use the same message-passing syntax.
  • Hybrid OOP (Java, C#): most things are objects, but primitive types (int, double, boolean in Java) are not objects. They get special treatment: no methods, no inheritance, stored by value. This improves performance but creates inconsistency.
  • OOP added to existing language (C++): objects and non-objects coexist completely. Entire programs can be written without any objects at all.
Pure OOP (Smalltalk/Ruby)
  • Conceptually clean — uniform treatment of all values
  • No special cases or exceptions to the object model
  • Maximally orthogonal
Hybrid OOP (Java primitives)
  • Primitives are more efficient (no heap allocation, no vtable)
  • Inconsistency: primitives cannot be used in generics directly
  • Autoboxing adds complexity and hidden overhead

Are Subclasses Also Subtypes? The Subtype Problem

Subtype
If type S is a subtype of type T, then any value of type S can be used wherever a value of type T is expected (Liskov Substitution Principle). The subtype can do everything the supertype does, and possibly more.

Are subclasses always subtypes? Ideally yes — a Dog should be usable anywhere an Animal is expected. But inheritance allows subclasses to weaken guarantees of the superclass:

Classic example: if Square extends Rectangle, and Rectangle has a setWidth() method, then calling setWidth() on a Square is problematic (a square must keep width == height). The Square is a subclass but not a proper subtype. This is the circle-ellipse problem or rectangle-square problem.

Languages address this with: abstract classes, interfaces (which define a true type contract), and final/sealed classes that cannot be subclassed.

Single vs Multiple Inheritance

Single Inheritance
Each class has at most one parent class. Simpler to implement; no ambiguity. Java (classes), C# (classes), Smalltalk.
Multiple Inheritance
A class can inherit from more than one parent class. More powerful (models real-world "is-a" in multiple dimensions) but introduces complications.

Diamond Problem

When class D inherits from both B and C, and both B and C inherit from A, which version of A's method does D use?

       A
      / \
     B   C
      \ /
       D   ← inherits A through both B and C — which A method wins?

Solutions:

  • C++: virtual base classes resolve diamond ambiguity; programmer must specify explicitly.
  • Java/C#: no multiple inheritance of classes (avoids diamond). Multiple interface implementation is allowed.
  • Python: MRO (Method Resolution Order) using C3 linearisation determines which parent's method is used.
  • Ruby: uses mixins (modules) instead of multiple class inheritance — a module can add methods to a class without the diamond problem.

Allocation and Deallocation of Objects

How are objects created and destroyed?

  • Heap allocation (Java, C#, Python, Ruby): objects are created on the heap with new or equivalent; garbage collection handles deallocation automatically. No dangling pointers, but unpredictable GC pauses.
  • Stack or heap (C++): objects can be created on the stack (automatic lifetime tied to scope) or the heap (manual new/delete). Programmer controls lifetime; risk of memory leaks and dangling pointers.
  • Constructors: special methods called when an object is created; initialise the object's state.
  • Destructors/Finalisers: called when an object is destroyed. C++ destructors are called deterministically; Java finalisers are called by the GC (non-deterministic). C# adds IDisposable/using for deterministic cleanup.

Nested Classes

Classes defined inside other classes. Reasons for nesting:

  • Logical grouping — the inner class only makes sense in the context of the outer class.
  • Encapsulation — the inner class can access private members of the outer class.
  • Reduced package namespace pollution.

Java distinguishes: static nested classes (no access to outer instance), inner classes (have access to outer instance), anonymous classes (one-off implementations), and local classes. C# nested classes are similar to Java's static nested classes.

Support for OOP in Smalltalk

Smalltalk is the canonical pure OOP language — everything is an object, computation happens through message passing.

Key characteristics

  • Everything is an object: integers (5), characters, classes, booleans, nil, and even blocks of code.
  • Message passing: methods are invoked by sending a message to an object. Syntax: receiver message or receiver message: argument.
  • Dynamic typing: no type declarations; any object can receive any message (if it doesn't understand, it raises a doesNotUnderstand error at runtime).
  • Single inheritance: each class has exactly one parent class. The root of the hierarchy is Object.
  • Classes are objects: classes are instances of metaclasses. This allows class-side methods (equivalent to static methods).
| x |
x := 5.                     "x = 5 (send := message)"
x printNl.                  "print x — sends printNl to integer object 5"
x class printNl.             "prints 'SmallInt' — class is an object too"
(x + 3) printNl.            "sends + message to 5 with arg 3 → 8"
(x > 3) ifTrue: ['yes'] ifFalse: ['no'].  "boolean is an object"

Support for OOP in C#

C# combines Java-like OOP (single class inheritance, multiple interface implementation) with important additions:

Properties

class BankAccount {
    private decimal balance;

    public decimal Balance {     // property with getter and setter
        get { return balance; }
        set {
            if (value < 0) throw new ArgumentException("Negative balance");
            balance = value;
        }
    }
}

BankAccount account = new BankAccount();
account.Balance = 100.0m;          // uses setter
decimal b = account.Balance;       // uses getter

Sealed classes

sealed prevents a class from being subclassed (like Java's final). Sealed classes enable certain compiler optimisations since dynamic dispatch is not needed.

Abstract classes

abstract class Shape {
    public abstract double Area();          // must be overridden
    public virtual string Describe() {      // may be overridden
        return $"Shape with area {Area()}";
    }
}

class Rectangle : Shape {
    double width, height;
    public Rectangle(double w, double h) { width = w; height = h; }
    public override double Area() { return width * height; }
}

Interfaces

interface IDrawable {
    void Draw();
    void Resize(double factor);
}

class Circle : Shape, IDrawable {
    public void Draw() { /* ... */ }
    public void Resize(double f) { /* ... */ }
    public override double Area() { return Math.PI * radius * radius; }
}

Support for OOP in Ruby

Ruby is a pure OOP language (like Smalltalk), but with a more conventional syntax (closer to Python/Perl).

Everything is an object

5.class          # => Integer
5.respond_to?(:+)  # => true
nil.class        # => NilClass
true.class       # => TrueClass

Classes and inheritance

class Animal
  def initialize(name)    # constructor
    @name = name          # instance variable (@ prefix)
  end

  def speak
    "..."
  end
end

class Dog < Animal        # single inheritance (< is "extends")
  def speak
    "Woof!"               # overrides Animal#speak
  end
end

d = Dog.new("Rex")
puts d.speak              # "Woof!" — dynamic dispatch

Mixins (Ruby's approach to multiple inheritance)

module Swimmable
  def swim
    puts "#{@name} is swimming"
  end
end

module Flyable
  def fly
    puts "#{@name} is flying"
  end
end

class Duck < Animal
  include Swimmable     # mixin
  include Flyable       # mixin — multiple capability without diamond problem
end

d = Duck.new("Donald")
d.swim   # from Swimmable module
d.fly    # from Flyable module
d.speak  # from Animal (or overridden)

Mixins allow Ruby to gain the benefits of multiple inheritance (reusing code from multiple sources) without the complications of the diamond problem. A module included in a class is inserted into the class's method lookup chain.

Access control in Ruby

  • public: accessible from anywhere (default)
  • private: only accessible within the class; cannot be called with an explicit receiver
  • protected: accessible from the class and its subclasses
Quick Check

A superclass Animal has a speak() method. A variable of type Animal is assigned a Dog object. When speak() is called on this variable, which version executes?

  • A Dog's speak() — dynamic binding uses the actual object type
  • B Animal's speak() — static binding uses the variable's declared type
  • C Both versions are called in sequence
  • D Compile error — cannot assign Dog to Animal variable

Chapter 12 Summary

  • Encapsulation: data + operations in one unit, public interface hides implementation. Access modifiers control visibility.
  • Inheritance: subclass inherits from superclass; can override methods. Enables code reuse.
  • Polymorphism: superclass reference can hold subclass object; correct method called at runtime.
  • Dynamic binding: method dispatched based on runtime type. C++ requires virtual; Java/C# use it by default.
  • Abstract classes: cannot be instantiated; declare abstract methods that subclasses must implement.
  • Interfaces: fully abstract types; a class can implement multiple interfaces (solving multiple-inheritance-of-type cleanly).
  • Exclusivity: Smalltalk/Ruby — pure OOP (everything is object). Java/C# — hybrid (primitives are not objects).
  • Diamond problem: multiple inheritance ambiguity. C++ uses virtual base classes; Java/C# prohibit multiple class inheritance; Ruby uses mixins.
  • Smalltalk: pure OOP, message passing, dynamic typing, classes are objects.
  • C#: Java-like OOP with properties, sealed classes, rich interface support, reified generics.
  • Ruby: pure OOP, mixins for multi-inheritance, everything is an object including primitives.
