Chapter 9

Subprograms

Subprograms are the fundamental abstraction mechanism in imperative languages. How parameters are passed, how closures capture scope, and how generics enable type-safe polymorphism are among the most exam-relevant topics in this course.

Fundamentals of Subprograms

Subprogram
A named block of code that can be called (invoked) from other parts of the program. Provides process abstraction — hiding the implementation of an operation behind a name and interface.

Each subprogram call activates a new activation of that subprogram. Key properties:

  • A subprogram has a single entry point (the beginning of its code).
  • The calling program is suspended during execution of the called subprogram.
  • Control always returns to the caller when the subprogram finishes.

Procedures vs Functions

ProcedureFunction
Return valueNo (or implicit)Yes — returns a single value
Called asStatementPart of an expression

In C and Java, all subprograms are syntactically functions (even those that return void). Ada distinguishes procedures from functions.

Formal vs Actual Parameters

Formal Parameters
The parameter names listed in the subprogram definition (also called the header). These are local variables within the subprogram.
Actual Parameters
The values or variables provided by the caller at the call site. Actual parameters are bound to formal parameters by a parameter-passing mechanism.
/* C — formal parameters are 'x' and 'y'; actual parameters at call site are 'a' and 'b' */
int add(int x, int y) { return x + y; }  /* x, y: formal parameters */
int result = add(a, b);                  /* a, b: actual parameters */

Positional vs Keyword Parameters

  • Positional: actual parameters are associated with formal parameters by their position. Most common.
  • Keyword: actual parameters are associated by name. Supported in Ada, Python, most modern languages. Allows any order and improves readability at the call site.
# Python — keyword arguments
def greet(name, greeting="Hello"):
    print(f"{greeting}, {name}!")

greet(greeting="Hi", name="Alice")  # keyword args, any order

Design Issues for Subprograms

  • What parameter-passing method(s) are provided?
  • Are parameter types checked?
  • Can subprograms be passed as parameters or returned as values?
  • Can subprograms be overloaded?
  • Can subprograms be generic?

Local Referencing Environments

Local variables of a subprogram are created when the subprogram is called and destroyed when it returns — they are stack-dynamic by default in most languages. This allows recursion: each call gets its own copy of local variables.

Some languages allow static local variables (e.g., static in C): the variable retains its value between calls but is local to the subprogram in scope. Useful for persistent state, but makes the function stateful and not referentially transparent.

Parameter-Passing Methods

⚠ Exam Focus — Very Important

Parameter passing methods are high-frequency exam material. For each method, understand: what is copied/shared, whether the caller's variable is changed, and which languages use it.

The key question is: when a subprogram is called, what is the connection between the actual parameter in the caller and the formal parameter in the subprogram?

Pass-by-Value

The value of the actual parameter is copied into the formal parameter. The subprogram works on its local copy; modifications do not affect the actual parameter.

/* C — pass-by-value: x is a copy of a */
void double_it(int x) {
    x = x * 2;  /* modifies only local copy */
}

int a = 5;
double_it(a);
/* a is still 5 — pass-by-value did not change it */

Advantage: safe — callee cannot accidentally corrupt caller's data.
Disadvantage: inefficient for large objects (entire value is copied). Also, the callee cannot return more than one value through parameters.

In Java, primitives are pass-by-value. Objects are pass-by-value of the reference (the reference is copied, not the object) — this means the callee can modify the object's fields but cannot redirect the caller's reference to point elsewhere.

Pass-by-Reference

The address (reference) of the actual parameter is passed. The formal parameter becomes an alias for the actual parameter — they refer to the same memory location. Any modification of the formal parameter changes the actual parameter.

/* C++ — pass-by-reference using & */
void double_it(int &x) {
    x = x * 2;  /* modifies the actual parameter */
}

int a = 5;
double_it(a);
/* a is now 10 */

Advantage: efficient for large objects (only address is passed); callee can return values through parameters.
Disadvantage: aliasing — the actual variable can be inadvertently modified. Reduces reliability; makes programs harder to reason about.

Pass-by-Value-Result (Copy-in / Copy-out)

A two-phase mechanism:

  1. On call: the value of the actual parameter is copied into the formal parameter (like pass-by-value).
  2. On return: the final value of the formal parameter is copied back into the actual parameter.
/* Ada conceptual example — in out parameter with value-result semantics */
procedure Increment(X : in out Integer) is
begin
    X := X + 1;
end;

A := 5;
Increment(A);
-- A is now 6

Difference from pass-by-reference: with value-result, updates to the formal parameter are not immediately visible through other aliases. Changes are only committed on return. With pass-by-reference, changes are immediately visible everywhere.

This distinction matters when the actual parameter is aliased (accessible through two different names). The two methods can produce different results in such cases.

Pass-by-Name

Used in ALGOL 60. The actual parameter text is substituted for every occurrence of the formal parameter in the subprogram body. The actual parameter expression is evaluated every time the formal parameter is used, in the environment of the caller.

/* ALGOL 60 conceptual: pass-by-name */
procedure swap(a, b):
    begin
        temp := a;
        a := b;
        b := temp;
    end

swap(i, arr[i])
/* substitutes as:
   temp := i;
   i := arr[i];      ← now i has changed!
   arr[i] := temp;  ← uses NEW value of i — assigns to wrong element! */

Pass-by-name can be surprising and error-prone (see the classic "swap with indexed variable" problem above). It was abandoned in practical languages after ALGOL 60 due to its unpredictable behaviour. However, it is interesting theoretically because it achieves "lazy evaluation" — the actual parameter is not evaluated until it is needed.

Parameters That Are Subprograms

Some languages allow subprograms to be passed as parameters to other subprograms. This requires the language to support higher-order functions or at least function pointers.

  • C: function pointers can be passed as parameters.
  • C++, Java, C#: lambda expressions / functional interfaces allow passing anonymous functions.
  • Python, Ruby, JavaScript: functions are first-class objects, passed naturally.

The referencing environment of the passed subprogram matters: should it use the environment where it was defined (static scoping — as in most languages) or where it was called (dynamic scoping)?

Overloaded Subprograms

Overloaded Subprogram
A subprogram that has the same name as another subprogram in the same scope, but with a different protocol (different number, type, or order of parameters). The compiler chooses the correct version at compile time based on the types of the actual parameters.
// Java — method overloading
void print(int n)    { System.out.println("int: " + n); }
void print(double d) { System.out.println("double: " + d); }
void print(String s) { System.out.println("string: " + s); }

print(42);       // calls print(int)
print(3.14);     // calls print(double)
print("hello");  // calls print(String)

Overloading improves writability: the caller uses a single, intuitive name regardless of the parameter type. Resolution is at compile time in statically typed languages.

C++ adds complexity: when multiple overloads match (via implicit conversions), the compiler applies a set of disambiguation rules.

Generic Subprograms

Generic Subprogram
A subprogram that is parameterised by type. A single generic definition can work with many different types while still being statically type-checked. Also called parametric polymorphism or templates.
⚠ Exam Focus — Know C++, Java, C# differences

The exam specifically asks about the differences between generics in C++, Java, and C#. Know the mechanism and key limitations of each.

Generics in C++ — Templates

C++ uses templates. The compiler instantiates a separate version of the function or class for each type it is used with. This is compile-time code generation.

// C++ template function
template <typename T>
T max(T a, T b) {
    return (a > b) ? a : b;
}

max(3, 7);        // T = int  — compiler generates int version
max(3.14, 2.71);  // T = double — compiler generates double version

Constraint: in original C++, there was no way to express type constraints — any type parameter could be used, and errors were only caught when the instantiation failed (often with cryptic error messages). C++20 introduces Concepts to address this.

Advantage: maximum performance — each instantiation is fully specialised, no runtime overhead.
Disadvantage: can lead to code bloat (separate compiled version for each type); complex error messages.

Generics in Java — Type Erasure

Java generics use type erasure. The generic type parameter exists at compile time for type checking, but is erased at runtime — only a single compiled version of the bytecode exists (parameterised by Object).

// Java generic method
public <T extends Comparable<T>> T max(T a, T b) {
    return a.compareTo(b) > 0 ? a : b;
}

Integer x = max(3, 7);        // T = Integer
String  s = max("apple", "banana");  // T = String

Type constraints: Java uses bounded type parameters (T extends SomeClass or T extends Interface) to restrict which types can be used. This is more expressive than raw C++ templates (which had no constraint syntax).

Limitation: due to type erasure, you cannot use type parameters in certain ways at runtime (e.g., cannot write new T() or T[10]). Cannot use primitive types as type parameters — must use wrappers (Integer, not int).

Wildcards: Java has a unique feature — wildcard type parameters (?), which allow flexible use of generic types in method signatures.

Generics in C# — Reified Generics

C# uses reified generics (also called first-class generics). Unlike Java's type erasure, C# preserves full type information at runtime. Each instantiation is a distinct type at the CLR level.

// C# generic method with constraint
public T Max<T>(T a, T b) where T : IComparable<T> {
    return a.CompareTo(b) > 0 ? a : b;
}

Max<int>(3, 7);
Max<string>("apple", "banana");

Constraints: C# uses where T : constraint syntax. Constraints can specify that T must implement an interface, must be a class or struct, must have a parameterless constructor (new()), etc. This is more expressive than Java's bounded wildcards.

Reification advantages: Can use new T() (if constrained with new()). Works with value types (no boxing). Runtime type checks work correctly. Reflection on generic types is possible.

C# allows generic classes, interfaces, delegates, and events — not just methods.

Closures

⚠ Exam Focus — Section 9.12 — Very Important

You must be able to understand JavaScript code involving closures, and provide the output of examples similar to the MakeAdder function. Understand how closures capture their environment.

Closure
A subprogram and the referencing environment in which it was defined, treated as a single unit. The closure "closes over" (captures) the variables from its defining scope, even after that scope has exited. When a nested function refers to a variable in an outer scope, that variable must be kept alive as part of the closure.

The MakeAdder Example (JavaScript)

function makeAdder(x) {
    // Returns a new function that adds x to its argument
    return function(y) {
        return x + y;  // 'x' is captured from the outer scope
    };
}

var add5  = makeAdder(5);   // x = 5 captured in closure
var add10 = makeAdder(10);  // x = 10 captured in a different closure

console.log(add5(3));   // 8   (5 + 3)
console.log(add10(3));  // 13  (10 + 3)
console.log(add5(7));   // 12  (5 + 7)

Key points:

  • makeAdder returns an inner function. When makeAdder(5) is called, it creates a closure that captures x = 5.
  • After makeAdder returns, its activation record is gone — but x is kept alive by the closure.
  • add5 and add10 are independent closures, each with their own captured x.
  • Calling add5(3) evaluates x + y = 5 + 3 = 8.

Closures in Python

def make_adder(x):
    def adder(y):
        return x + y  # x captured from enclosing scope
    return adder

add3 = make_adder(3)
print(add3(10))  # 13
print(add3(20))  # 23

Closures sharing mutable state

function makeCounter() {
    var count = 0;           // captured mutable variable
    return {
        increment: function() { count += 1; },
        value:     function() { return count; }
    };
}

var c = makeCounter();
c.increment();
c.increment();
console.log(c.value());  // 2 — both functions share the same 'count'

Both returned functions close over the same count variable. Mutations are shared. This is a powerful pattern for encapsulating private state.

Implementation requirement

Closures require that free variables (captured variables) be stored on the heap, not the stack, because they must outlive the activating call. This is why closures are not trivially available in C (no heap allocation for stack frames by default). In languages like Python, JavaScript, Java 8+, and C#, the runtime handles this automatically.

Coroutines

⚠ Exam — Be able to provide output from pseudocode

The exam may show pseudocode involving coroutines and ask you to trace the output. Understand the execution model: two (or more) routines take turns executing, control switches explicitly.

Coroutine
A subprogram that has multiple entry points and that maintains its state between invocations. Unlike normal subprograms (which always start at the beginning), a coroutine resumes from where it last suspended. Control is passed back and forth between coroutines using a resume operation; there is no caller/callee hierarchy — they are symmetric.

Key differences from normal subprograms

Normal SubprogramCoroutine
Entry pointsOne (always starts at beginning)Multiple (resumes from last yield point)
State between callsLost (local variables re-created)Preserved (execution state maintained)
Caller-calleeAsymmetric hierarchySymmetric — either can resume the other
ConcurrencyNot inherentCooperative concurrency (not preemptive)

Pseudocode example (tracing)

/* Pseudocode — two coroutines taking turns */
coroutine A:
    print "A1"
    resume B
    print "A2"
    resume B
    print "A3"

coroutine B:
    print "B1"
    resume A
    print "B2"
    resume A

/* Execution trace starting with A: */
-- A starts
A1
-- A resumes B
B1
-- B resumes A (B suspends, A continues from after "resume B")
A2
-- A resumes B (A suspends, B continues from after "resume A")
B2
-- B resumes A (B suspends, A continues from after "resume B")
A3
-- A finishes

Output: A1 B1 A2 B2 A3

Real-world use

  • Generators in Python (yield), JavaScript (function*), and C# (yield return) are restricted coroutines that produce a sequence of values lazily.
  • async/await in Python, C#, JavaScript is built on coroutine-like execution.
  • SIMULA 67 was one of the first languages with coroutines. Lua has first-class coroutines.
Quick Check

Consider this JavaScript code:

function outer(n) {
  return function(x) { return n * x; };
}
var triple = outer(3);
console.log(triple(4));

What is printed?

  • A 12
  • B 7
  • C 4
  • D ReferenceError — n is not defined

Chapter 9 Summary

  • Formal vs actual parameters: formal parameters are in the definition; actual parameters are at the call site.
  • Pass-by-value: copy of value passed; caller's variable not modified. Used in C (default), Java (primitives).
  • Pass-by-reference: address passed; formal is alias for actual; caller's variable can be modified. C++ (&), Pascal (var).
  • Pass-by-value-result: copy in, copy back on return. Different from pass-by-reference when aliasing exists.
  • Pass-by-name: ALGOL 60; actual expression substituted textually; evaluated in caller's environment; error-prone.
  • Overloading: same name, different protocols. Resolved at compile time.
  • C++ templates: compile-time instantiation; one version per type; no constraints until C++20.
  • Java generics: type erasure; single bytecode; no primitives; bounded wildcards for constraints.
  • C# generics: reified; runtime type info preserved; rich where constraints; works with value types.
  • Closures: subprogram + captured environment. Captured variables live on the heap. MakeAdder pattern.
  • Coroutines: symmetric, multiple entry points, state preserved between calls. Execution alternates via resume.
