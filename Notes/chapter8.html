Chapter 8

Statement-Level Control Structures

Selecting which statements execute, and how many times. From simple if-else to Dijkstra's non-deterministic guarded commands — control flow design has profound implications for correctness.

Selection Iteration for-each Ruby Blocks Guarded Commands Non-determinism

Selection Statements

A selection statement (also called conditional statement) selects between two or more execution paths based on a Boolean condition or expression value.

Two-Way Selection: if-then-else

The most fundamental control structure. Design issues include:

  • What can appear in the condition? (Boolean only, or any type?)
  • What is the clause syntax? (single statement, block, or any?)
  • How is the "dangling else" problem resolved?

The Dangling else Problem

/* C — which 'if' does the 'else' belong to? */
if (x > 0)
    if (y > 0)
        z = 1;
else       /* Belongs to inner 'if', not outer! */
    z = -1;

Most languages (C, Java, C++) resolve this by the nearest unmatched if rule: an else matches the nearest preceding if that has no else. This is often a surprise to programmers. Solutions:

  • Explicit delimiters: Pascal uses begin...end; Scheme and LISP use parentheses.
  • Required else: never have dangling ifs.
  • Mandatory blocks: Python and Ruby require indentation/keyword delimiters, so the structure is unambiguous.
  • Special keyword: Ada uses end if to close every if.

Selector Expressions (Conditional Expressions)

In some languages, selection is an expression that returns a value:

/* C/C++/Java — ternary operator */
int max = (a > b) ? a : b;

/* Python — conditional expression */
max_val = a if a > b else b

Multiple Selection: case / switch

Selects one of several paths based on the value of an expression (the selector).

C-style switch

/* C — fall-through semantics by default */
switch (grade) {
    case 'A':
        printf("Excellent\n");
        break;       /* break required to prevent fall-through */
    case 'B':
    case 'C':
        printf("Passing\n");
        break;
    default:
        printf("Failing\n");
}

The fall-through default of C's switch is a common source of bugs — forgetting a break causes execution to continue into the next case. Languages differ in how they handle this:

  • Java: same fall-through as C, but also has the new switch expression (Java 14+) which does not fall through.
  • Ada: no fall-through; all choices are exclusive and the compiler warns if a value is not covered.
  • Ruby: uses case...when with no fall-through; can match on ranges and patterns.
  • Python: the structural pattern matching (match...case, Python 3.10+) has no fall-through.

Design issue: what can be the selector?

C's switch requires an integer-compatible type. C++ allows enumerations. Java 7+ allows strings. Ada allows any discrete type. Ruby allows any object with an equality test.

Iterative Statements

An iterative statement (loop) causes one or more statements to execute zero or more times.

Counter-Controlled Loops

A loop that executes a fixed number of times, controlled by a counter variable.

FORTRAN-style do loop

The earliest DO loops had a fixed loop variable, initial value, final value, and step. Modifying the loop variable inside the loop was undefined or forbidden in some languages.

C-style for loop

/* C — flexible: initialisation, condition, update are arbitrary expressions */
for (int i = 0; i < 10; i++) { ... }

/* Can use any expressions — not just counter control */
for (p = head; p != NULL; p = p->next) { ... }  /* linked list traversal */

The C for is extremely flexible but is fundamentally a "while loop with initialiser and update" — not specifically a counter-controlled loop. This flexibility is expressive but can be misused.

Python range-based for

# Python — counter-controlled using range()
for i in range(10):
    print(i)

for i in range(0, 10, 2):  # range(start, stop, step)
    print(i)  # 0, 2, 4, 6, 8

Logically-Controlled Loops

Loops that continue executing while (or until) a condition is true. Not counter-controlled — the number of iterations depends on program state.

Pre-test loop (while)

/* Condition checked BEFORE each iteration — may execute zero times */
while (condition) {
    body;
}

Post-test loop (do-while / repeat-until)

/* Condition checked AFTER each iteration — executes at least once */
do {
    body;
} while (condition);  /* C/Java */

repeat
    body
until condition;      /* Pascal/Ada */

Note: C's do-while continues while the condition is true. Pascal's repeat-until continues until the condition is true (i.e., exits when condition becomes true). The semantics are inverted — a common source of confusion.

Mid-test loops

Ada and some other languages allow an exit when inside a loop, which effectively tests the condition in the middle of the loop body. This avoids the "loop-and-a-half" problem of duplicating code for the first check before the loop.

Iterator-Based Loops (for-each)

Iterates over every element in a collection without the programmer managing indices or pointers.

// Java — enhanced for (for-each)
for (String name : nameList) {
    System.out.println(name);
}
# Python — all for loops are iterator-based
for element in my_list:
    print(element)

for key, value in my_dict.items():
    print(key, value)
# Ruby — multiple iterator methods
[1, 2, 3].each { |x| puts x }
(1..5).each { |n| puts n }

Iterator-based loops improve readability and reliability: no off-by-one errors, no accidental index mutation, works naturally with any collection type that supports iteration.

Ruby Blocks

⚠ Exam Focus — Section 8.3.4

Ruby blocks are specifically highlighted in the scope. Understand the syntax, how blocks are passed to methods, and how they differ from a language that separates iteration from the iteration logic.

In Ruby, a block is an anonymous chunk of code that can be passed to a method and invoked by that method. Blocks are closures — they capture the surrounding scope.

Block syntax

# Brace syntax (single-line, conventional)
[1, 2, 3].each { |x| puts x }

# do...end syntax (multi-line, conventional)
[1, 2, 3].each do |x|
  puts x * 2
end

# Block with the times method
5.times { |i| puts i }    # 0 1 2 3 4

# Block with upto
1.upto(5) { |n| puts n }  # 1 2 3 4 5

# collect/map — returns new array
doubled = [1, 2, 3].collect { |x| x * 2 }  # [2, 4, 6]

# select — returns elements for which block is true
evens = [1, 2, 3, 4, 5].select { |x| x.even? }  # [2, 4]

# inject/reduce — accumulator
sum = [1, 2, 3, 4].inject(0) { |acc, x| acc + x }  # 10

Why Ruby blocks matter conceptually

Ruby's design turns control structures (iteration, resource management, error handling) into method calls that accept blocks. This is more flexible than built-in syntax because:

  • Any object can define its own iteration behaviour (by implementing each).
  • The programmer does not need to learn a new syntax for each kind of iteration.
  • Blocks are first-class closures — they capture local variables from the enclosing scope.
# Resource management with blocks (ensures file is closed)
File.open("data.txt") do |file|
  file.each_line { |line| puts line }
end  # file is automatically closed when the block exits

Guarded Commands (Dijkstra, 1975)

⚠ Exam Focus — Section 8.5

Guarded commands are a theoretical concept with practical implications for program correctness. Understand the syntax, the non-determinism, and the rationale for their design. They appear regularly in exams.

Edsger Dijkstra proposed guarded commands as a foundation for program verification (proving programs correct). The key insight is that if there is no reason to prefer one execution path over another, the language should not force one. This is called non-determinism.

Guarded Selection Command

Syntax (pseudo-code):

if <guard1> → <statement_list1>
[] <guard2> → <statement_list2>
...
[] <guardn> → <statement_listn>
fi

Semantics:

  • Each guard is a Boolean expression.
  • All guards are evaluated.
  • If none of the guards are true → error (program terminates with an error — it is the programmer's responsibility to ensure at least one guard is true).
  • If one or more guards are true → one is selected non-deterministically and its statement list is executed.

Example: maximum of two values

if x >= y → max := x
[] y >= x → max := y
fi

When x == y, both guards are true. Either branch gives the correct result. The language need not choose a particular one — and this non-determinism explicitly documents that both answers are equivalent.

Contrast with if-else

In a traditional if-else, even when both branches are semantically equivalent (as in the example above), the programmer must arbitrarily pick one ordering. The guarded command expresses the symmetry explicitly.

Guarded Loop Command

do <guard1> → <statement_list1>
[] <guard2> → <statement_list2>
...
[] <guardn> → <statement_listn>
od

Semantics:

  • As long as at least one guard is true, evaluate all guards and non-deterministically execute one of the true guards' statement lists.
  • When all guards are false, the loop terminates.

Example: sort two variables

do x > y → temp := x; x := y; y := temp
od

This loop continues swapping x and y as long as x > y. When x ≤ y, the loop terminates with x holding the smaller value.

Example: non-deterministic sort with two variables

do x > y → x := x - y
[] y > x → y := y - x
od
/* when x == y, both guards false, loop ends — this computes GCD! */

Significance of guarded commands

  • Program verification: the explicit guards make pre- and post-conditions for each branch very clear, supporting formal proof of correctness.
  • Non-determinism: allows expressing algorithms where the order of operations genuinely does not matter, making the algorithm more abstract and general.
  • Influence: guarded commands influenced Ada's select statement for concurrent programming, and various verification languages (TLA+, B, Z). They do not appear in mainstream imperative languages but are important theoretically.
Quick Check

In Dijkstra's guarded selection command, what happens if more than one guard evaluates to true?

  • A One of the true guards is selected non-deterministically
  • B A runtime error is raised
  • C All true guards' statement lists are executed in order
  • D The first true guard in the list is selected

Chapter 8 Summary

  • Dangling else: resolved by nearest-unmatched-if rule in most languages. Use explicit delimiters to prevent ambiguity.
  • switch/case: C has fall-through (requires break); Ada does not. Selector types vary by language.
  • Counter loops: C's for is very flexible; Ada/Pascal's for is restricted to integer ranges.
  • while (pre-test): may execute zero times. do-while (post-test): always executes at least once.
  • for-each / iterator: safer, more readable iteration over collections. Avoids index errors.
  • Ruby blocks: closures passed to methods; used for iteration, resource management. All iteration in Ruby is via blocks and methods.
  • Guarded commands: guards evaluated, one true guard selected non-deterministically. Error if no guard is true. Error does NOT occur if multiple guards are true. Supports program verification and non-determinism.
