Chapter 5

Names, Bindings & Scopes

Variables as abstractions of memory. How attributes are bound to entities, when that binding happens, and how names are resolved in different languages.

Names Variables Binding Time Static / Dynamic Type Storage Lifetime Static Scope Dynamic Scope

Names (Identifiers)

Imperative languages are abstractions of the von Neumann architecture (memory + processor). Variables are abstractions of memory cells. The design of variables raises issues around names, bindings, and scopes.

Form of a Name

Names have three design dimensions: length, use of special characters, and case sensitivity.

LanguageName Length Rule
Fortran IMax 6 characters
Fortran 95Max 31 characters
C99No limit; only first 63 significant (external names: 31)
C++, C#, Ada, JavaNo limit, all significant

Special Character Conventions

  • PHP: All variable names begin with $
  • Perl: Scalars use $, arrays use @, hashes use %
  • Ruby: Instance variables begin with @; class variables with @@
📝 Case Sensitivity

C-based languages are case-sensitive — myname ≠ myName. This hurts readability. Mixed-case predefined names like IndexOutOfBoundsException also hurt writability. Many non-C languages are not case-sensitive.

Reserved Words vs Keywords

Reserved word
Cannot be used as a user-defined identifier. E.g., int in C always means integer type. Too many reserved words = name collision problems (COBOL has ~300).
Keyword
Only special in certain contexts. E.g., in Fortran, Real is a type name in Real VarName but can be a variable name in Real = 3.4.

Variables

A variable is an abstraction of a memory cell, characterized by six attributes:

AttributeDescription
NameThe identifier (not all variables have one — anonymous heap vars)
AddressThe memory address. Can change between executions. Two names for same address = aliases.
ValueContents of the memory cell. One abstract cell even if many bytes (e.g., float = 4 bytes = 1 abstract cell).
TypeDefines the set of valid values, operations, and storage representation.
LifetimeTime during which the variable is bound to a memory cell.
ScopeRange of statements over which the variable is visible.
⚠ Aliases

Aliases — when two names refer to the same memory cell — are bad for readability because changing one variable unexpectedly changes another. Created by pointers, references, and unions.

The Concept of Binding

Binding
An association between an attribute and an entity. E.g., variable ↔ type, variable ↔ value, operator ↔ symbol.
Binding time
The time at which the binding takes place.
Binding TimeExample
Language design timeBinding * to multiplication
Language implementation timeBinding float to a 32-bit IEEE representation
Compile timeBinding a variable to a type in C or Java
Load timeBinding a C static local variable to a memory cell
RuntimeBinding a non-static local variable to a memory cell
✓ Static Binding
  • Occurs before runtime
  • Remains unchanged throughout execution
  • Enables compile-time type checking
  • More efficient (no runtime overhead)
~ Dynamic Binding
  • Occurs during execution
  • Can change during execution
  • More flexible / generic code
  • Higher cost; harder to detect errors

Type Binding

Static Type Binding

A program statement used to declare types before use.

int myVar;       // C++ explicit declaration
double salary;   // type bound at compile time

Type inferred from naming conventions or first appearance. Used in BASIC, Perl, Ruby, PHP.

✓ Advantage
  • Minor writability convenience
✗ Disadvantage
  • Typographic errors can't be caught by the compiler — a misspelled name creates a new variable silently.

Type determined from context. Used in C#, Visual BASIC 9+, Haskell, ML, F#.

var value = 12;     // C# — value is inferred as int
var name  = "hi";  // inferred as string

The type is still static — it's just not written explicitly. The binding still happens at compile time.

Dynamic Type Binding

Type is determined at runtime via assignment. Variables can hold different types at different times.

// JavaScript — dynamic type binding
let list = [2, 4.33, 6, 8];  // list is now an array
list = 17.3;                   // list is now a float — no error!
✓ Advantages
  • Flexibility — variable can hold any type
  • Enables generic program units
✗ Disadvantages
  • Runtime type checking overhead
  • Type errors are harder to detect
  • Less reliable programs

Storage Bindings & Lifetimes

Lifetime = the time during which a variable is bound to a memory cell. Allocation = getting a cell from available memory. Deallocation = returning it to the pool.

Bound to memory before execution begins; remain bound throughout the entire program run.

int f() {
    static int sVar = 10;   // allocated once, persists
    return ++sVar;
}
✓ Advantages
  • Efficient — direct addressing
  • History-sensitive subprograms
✗ Disadvantages
  • No recursion support
  • Storage cannot be shared

Storage bound when declaration is elaborated; deallocated at end of scope. The normal case for local variables in C, Java, etc.

void f() {
    int myVar;   // allocated on call, freed on return
}
✓ Advantages
  • Allows recursion
  • Conserves storage — reused across calls
✗ Disadvantages
  • Allocation/deallocation overhead
  • Not history-sensitive
  • Indirect addressing — less efficient

Allocated and deallocated by explicit programmer directives. Accessed only via pointers/references. All Java objects; new/delete in C++.

// C++ — programmer controls allocation
int* p = new int(12);
delete p;
✓ Advantages
  • Custom dynamic storage management
✗ Disadvantages
  • Unreliable — dangling pointers, memory leaks
  • Slow (heap management overhead)

Allocation/deallocation caused by assignment statements. All variables in APL; strings and arrays in Perl, JavaScript, PHP.

// JavaScript — implicit heap dynamic
let arr = [1, 2, 3];    // allocated on assignment
arr = "now a string";   // old array freed, new string allocated
✓ Advantages
  • Maximum flexibility
✗ Disadvantages
  • Most attributes bound dynamically — very inefficient
  • Loss of compile-time error detection

Scope

The scope of a variable is the range of statements over which it is visible. Local variables are declared in the unit; nonlocal variables are visible but declared elsewhere.

Static Scope

Based on program text. To find a name, search declarations starting in the local unit and expanding outward through enclosing scopes until found. Enclosing static scopes are called static ancestors; the nearest is the static parent. 

// Static scope example — C
void f() {
    int i = 10;        // outer i
    for (...) {
        int i = 20;    // inner i — HIDES outer i
        // outer i is inaccessible here in C/C++
    }
    // outer i is visible here again
}
📝 Variable Hiding Rules by Language
  • C, C++: Variable hiding in nested blocks is legal.
  • Java, C#: Variable hiding in blocks is not allowed — considered error-prone.

Global Scope

  • C, C++, PHP, Python: Global variables declared outside functions; scope covers all functions.
  • PHP: Globals are accessible inside functions only via $GLOBALS['x'] or a global $x; declaration.
  • Python: A global can be read inside a function; to assign to it, you must declare global x inside the function.

The let Construct

Functional languages often create local scope with a let construct instead of statement blocks. The bindings behave like local named constants inside a limited expression scope. 

// F#-style idea
let radius = 5
let area = 3.14159 * radius * radius

Declaration Order

LanguageRule
C89Declarations must appear at the beginning of a function/block
C99, C++, Java, JavaScript, C#Declarations can appear anywhere a statement can
C++, Java, C#for-loop control variables are scoped to the loop body
JavaScriptTraditional function-scoped locals can be referenced before the declaration statement is reached
C# Scope Quirk

In C#, a block variable's scope is considered the whole block even when the declaration appears later. You still cannot use it before the declaration line, but its name is already reserved for that block.

Dynamic Scope

Based on calling sequences at runtime. To find a variable, search back through the call chain. This is temporal, not spatial. 

Big {
    declare X
    Sub1 {
        declare X        // local X in Sub1
        call Sub2
    }
    Sub2 {
        refer to X       // which X?
    }
    call Sub1
}

// Static  scope: X in Big  (search text structure)
// Dynamic scope: X in Sub1 (search call chain)
✓ Advantage
  • Convenient — eliminates the need for explicit parameters in some cases
✗ Disadvantages
  • Executing subprogram's locals are visible to all called subprograms
  • Impossible to statically type-check
  • Hard to read — must trace call chain to understand variable reference
  • Slower access to non-local variables

Scope vs Lifetime

These are related but different concepts:

VariableScopeLifetime
Local var in Java methodMethod body onlySame as scope
C static local varInside function onlyEntire program execution

The C static example is the key one: narrow scope, wide lifetime. The variable is alive the whole program but only nameable within the function.

Referencing Environment

The referencing environment of a statement is the full collection of names visible at that point.

  • In a static-scoped language: local names plus visible names from static ancestor scopes.
  • In a dynamic-scoped language: local names plus names from all currently active subprograms on the call stack.

Named Constants

A named constant is a variable bound to a value only once. Improves readability and maintainability.

const int SIZE = 10;   // C++
int arr[SIZE];
for (int i = 0; i < SIZE; i++) { ... }
LanguageBinding
Fortran 95Constant-valued expressions only
Ada, C++, JavaExpressions of any kind (can be dynamic)
C# constCompile-time bound (manifest constants)
C# readonlyDynamically bound (runtime)

Chapter 5 Summary

  • Binding time: Earlier = more efficient, less flexible. Later = more flexible, higher cost.
  • Static type binding: Via explicit declaration, implicit declaration, or type inferencing. Type is bound at compile time.
  • Dynamic type binding: Via assignment. Used in Python, JavaScript, Ruby. Flexible but error-prone.
  • 4 storage lifetime categories: static → stack-dynamic → explicit heap-dynamic → implicit heap-dynamic (ordered from most to least control).
  • Static scoping: Based on text structure. More readable and reliable. Most modern languages use this.
  • Dynamic scoping: Based on call chain. More convenient but harder to reason about. Used in some older/scripting languages.
  • Scope ≠ Lifetime: A C static local is the canonical example of narrow scope + wide lifetime.
