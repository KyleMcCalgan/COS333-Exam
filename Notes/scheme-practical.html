Scheme in Practice

Every exam since 2022 has included a 5-mark Scheme coding question. They all follow a single formula: write a recursive function that processes a list of integers using cond. This page reverse-engineers that formula from every past paper.

cond template car / cdr / cons 4 archetypes 2022–2025 exams Decision process
Exam Pattern (5 marks)

The Scheme question on the exam is always 5 marks and always asks you to write a recursive function over a list of integers. The question will specify behaviour per element — what to keep, what to discard, what to transform. It is always solvable with the same 3-clause cond skeleton. Master that skeleton and the variants, and this question is free marks.

Mental Model

Scheme is a dialect of Lisp. Everything in Scheme is either an atom (a number, symbol, boolean) or a list (a pair of elements enclosed in parentheses). All code is data and all data looks like code — this is what S-expression syntax means.

Key shift

In Scheme, f(x, y) is written (f x y). The function comes first, inside the parentheses. There are no infix operators — 2 + 3 is (+ 2 3). Once you internalise this, reading Scheme becomes mechanical.

List Anatomy

A Scheme list is a chain of cons pairs. Each pair holds one element (the car) and a pointer to the rest (the cdr). The last cdr is the empty list '().

'(3 -1 5 0) ← this list looks like this internally: (cons 3 (cons -1 (cons 5 (cons 0 '())))) (car '(3 -1 5 0)) → 3 ← the first element (cdr '(3 -1 5 0)) → '(-1 5 0) ← everything else (cons 3 '(-1 5)) → '(3 -1 5) ← prepend 3
The recursion mental model

Every exam question asks you to process a list element by element. The strategy is always the same: handle the head (car), then recurse on the tail (cdr). You never need a loop — the recursion handles iteration.

The cond Template

Every exam Scheme answer uses cond with exactly two or three clauses. Here is the skeleton — memorise this:

(define (functionName lst) (cond ((null? lst) '()) ← base case: empty list → empty list ((condition (car lst)) ← does the head satisfy the condition? (cons ??? (functionName (cdr lst)))) ← yes: include something + recurse (else ← head does NOT satisfy condition (cons ??? (functionName (cdr lst))))) ← no: include something different + recurse )

The two decisions you make are:

  1. What is the condition on (car lst)? — Positive? Odd? Zero? A combination?
  2. What do you put in each cons? — The head unchanged? The head transformed? Nothing (skip it)?

When you skip an element (don't include it in the result), you drop the cons and just return the recursive call directly:

Including the head: (cons (car lst) (fn (cdr lst))) Transforming the head: (cons (* (car lst) 2) (fn (cdr lst))) ← double it Skipping the head: (fn (cdr lst)) ← no cons, just recurse

The Exam Formula — What Changes Each Year

Looking at every exam from 2022–2025, the question always specifies behaviour for elements in a list. The formula has exactly three moving parts:

Year Function name Condition When true: do this When false: do this
2022 getPositiveOddValues (> x 0) AND (odd? x) Keep (cons it) Skip (drop it)
2023 doublePositives (> x 0) Double it — (* x 2) Keep unchanged
2024 removePositiveOddValues (> x 0) AND (odd? x) Skip (drop it) Keep (cons it)
2025 getZerosDoublePositives Three-way: zero / positive / negative Keep zero; double positive Skip negatives

Notice: 2022 and 2024 are the same condition but with swapped actions — one keeps matching elements, one discards them. 2025 adds a three-way split (zero / positive / negative). These are your four exam archetypes.

The Four Archetypes

Every possible exam question is a combination of these patterns. Know all four cold.

Archetype 1 · Filter-Keep

Return only elements that match the condition. Drop everything else.

(define (fn lst)
  (cond
    ((null? lst) '())
    ((condition (car lst))
     (cons (car lst)
           (fn (cdr lst))))
    (else
     (fn (cdr lst)))))
Archetype 2 · Filter-Drop

Return only elements that do NOT match. Drop matching ones.

(define (fn lst)
  (cond
    ((null? lst) '())
    ((condition (car lst))
     (fn (cdr lst)))
    (else
     (cons (car lst)
           (fn (cdr lst))))))
Archetype 3 · Transform-or-Keep

Transform matching elements; keep others unchanged. Nothing is dropped.

(define (fn lst)
  (cond
    ((null? lst) '())
    ((condition (car lst))
     (cons (transform (car lst))
           (fn (cdr lst))))
    (else
     (cons (car lst)
           (fn (cdr lst))))))
Archetype 4 · Three-Way Split

Three distinct cases (zero / positive / negative). Needs three cond clauses plus base.

(define (fn lst)
  (cond
    ((null? lst) '())
    ((zero? (car lst))
     (cons 0 (fn (cdr lst))))
    ((> (car lst) 0)
     (cons (transform (car lst))
           (fn (cdr lst))))
    (else
     (fn (cdr lst)))))

Decision Process — Reading an Exam Question

When you see the Scheme question, work through this checklist:

1
Write the skeleton immediately. Start with (define (name lst) (cond ((null? lst) '()) … )). Don't stare at a blank screen.
2
Identify how many distinct cases the question describes. Two cases → two cond branches. Three cases (e.g. zero / positive / negative) → three branches.
3
For each case, decide: keep, transform, or drop?
  • Keep unchanged → (cons (car lst) (fn (cdr lst)))
  • Transform → (cons (f (car lst)) (fn (cdr lst)))
  • Drop/skip → (fn (cdr lst)) — no cons
4
Write the condition for each branch using (car lst). Positive: (> (car lst) 0). Negative: (< (car lst) 0). Both positive and odd: (and (> (car lst) 0) (odd? (car lst))).
5
Make the last condition else, never a specific check. The else catches everything not matched by previous branches. Never repeat a positive check in the else — if you've already handled positive, else covers zero and negative automatically.
6
Trace through a tiny example mentally. Try '(2 -1 3) or '(0 3 -2) — two elements you expect to appear in the output and one you expect to be dropped or transformed. Check your output matches the question description.

Past Exam Walkthroughs

Filter-Keep getPositiveOddValues(lst) 2022 Exam · Q4 · 5 marks

Question

Write a Scheme function getPositiveOddValues that accepts a list of integers and returns a new list containing only those elements that are both positive and odd.

Example: (getPositiveOddValues '(3 -1 4 7 -2 5)) → '(3 7 5)

Analysis

Archetype: Filter-Keep. Two cases. Condition: (> x 0) AND (odd? x). If true → keep. If false → skip. There are no transforms.

Solution

(define (getPositiveOddValues lst)
  (cond
    ((null? lst) '())
    ((and (> (car lst) 0) (odd? (car lst)))
     (cons (car lst) (getPositiveOddValues (cdr lst))))
    (else
     (getPositiveOddValues (cdr lst)))))

Trace — (getPositiveOddValues '(3 -1 4))

call getPositiveOddValues '(3 -1 4) → (car lst) = 3 ; (> 3 0) = #t, (odd? 3) = #t → keep call getPositiveOddValues '(-1 4) → (car lst) = -1 ; (> -1 0) = #f → skip (else) call getPositiveOddValues '(4) → (car lst) = 4 ; (odd? 4) = #f → skip (else) call getPositiveOddValues '() return '() return '() ; skip 4, return unchanged recursive result return '() ; skip -1 return '(3) ; cons 3 onto '() → '(3)
Transform-or-Keep doublePositives(lst) 2023 Exam · Q4 · 5 marks

Question

Write a Scheme function doublePositives that accepts a list of integers and returns a new list where each positive element is doubled and all other elements are kept unchanged.

Example: (doublePositives '(3 -1 0 4)) → '(6 -1 0 8)

Analysis

Archetype: Transform-or-Keep. Two cases. Condition: (> x 0). If true → double ((* x 2)). If false → keep unchanged. Nothing is dropped.

Solution

(define (doublePositives lst)
  (cond
    ((null? lst) '())
    ((> (car lst) 0)
     (cons (* (car lst) 2) (doublePositives (cdr lst))))
    (else
     (cons (car lst) (doublePositives (cdr lst))))))

Key difference from 2022

The else branch now has a cons — because non-positive elements are kept, not dropped. In 2022, the else was just the bare recursive call with no cons because non-matching elements were dropped.

Filter-Drop removePositiveOddValues(lst) 2024 Exam · Q4 · 5 marks

Question

Write a Scheme function removePositiveOddValues that accepts a list of integers and returns a new list with all elements that are both positive and odd removed.

Example: (removePositiveOddValues '(3 -1 4 7 -2 5)) → '(-1 4 -2)

Analysis

Archetype: Filter-Drop. The condition is identical to 2022, but the actions are swapped — matching elements are now dropped, non-matching elements are kept. This is the classic "get vs remove" variant.

Solution

(define (removePositiveOddValues lst)
  (cond
    ((null? lst) '())
    ((and (> (car lst) 0) (odd? (car lst)))
     (removePositiveOddValues (cdr lst)))
    (else
     (cons (car lst) (removePositiveOddValues (cdr lst))))))

The swap pattern

Compare with 2022: the two body expressions are exactly swapped. In 2022, the matching branch has cons and the else is bare. Here, the matching branch is bare and else has cons. If you see "remove" or "without" in the question, swap the branches.

Three-Way Split getZerosDoublePositives(lst) 2025 Exam · Q4 · 5 marks

Question

Write a Scheme function getZerosDoublePositives that accepts a list of integers and returns a new list where: zeros are kept, positives are doubled, and negatives are removed.

Example: (getZerosDoublePositives '(3 0 -2 1 -5 0)) → '(6 0 2 0)

Analysis

Archetype: Three-Way Split. Three distinct behaviours require three cond clauses (plus the base). The trick is ordering: check zero? first, then > 0, then else catches negatives.

Solution

(define (getZerosDoublePositives lst)
  (cond
    ((null? lst) '())
    ((zero? (car lst))
     (cons 0 (getZerosDoublePositives (cdr lst))))
    ((> (car lst) 0)
     (cons (* (car lst) 2) (getZerosDoublePositives (cdr lst))))
    (else
     (getZerosDoublePositives (cdr lst)))))

Why check zero first?

If you checked (> x 0) first, zeros fall through to else and get dropped — wrong. The more specific case (zero?) must precede the more general case (> 0). Order your cond branches from most specific to least specific.

Common mistakes

Mistake 1 — Forgetting the base case

Every recursive function needs ((null? lst) '()) as the first clause. Without it, the function will throw an error when it reaches the end of the list because it tries to call (car '()).

Mistake 2 — Using the wrong argument for recursion

The recursive call is always on (cdr lst), never on lst (that causes infinite recursion) and never on (car lst) (that's an atom, not a list). Every branch ends with (functionName (cdr lst)).

Mistake 3 — Wrapping the condition in an extra list

The condition in a cond clause is just an expression: ((> (car lst) 0) …). Do not write ((list (> (car lst) 0)) …) or any variant. The condition is tested directly as a boolean.

Mistake 4 — Putting cons where you mean to skip

When you want to drop an element, the branch body is just (fn (cdr lst)) — the recursive call alone. If you accidentally write (cons (car lst) (fn (cdr lst))), the element is kept instead of dropped. Double-check which branches have cons and which don't.

DrRacket note

The exam environment is DrRacket with #lang racket. All the forms above work identically in that environment. The 2023 exam explicitly required the #lang racket directive on the first line — write it by default. You can use define, cond, car, cdr, cons, null?, odd?, even?, zero?, and all arithmetic without any imports.
