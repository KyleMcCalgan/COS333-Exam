Chapter 6

Data Types

What data types are, how they're designed across languages, and the trade-offs between type safety, flexibility, and performance.

Primitive Types Arrays Records Unions Pointers Type Checking Strong Typing Coercion
Data Type
A collection of data values with a range of values, a way to store them in memory, and a set of predefined operations on those values.

Primitive Data Types

Primitive types are not defined in terms of other types. Some are direct hardware reflections; others require a little non-hardware support.

May be as many as 8 different sizes. Java's signed integers:

TypeSizeRange
byte8-bit−128 to 127
short16-bit−32,768 to 32,767
int32-bit~−2.1B to 2.1B
long64-bit~−9.2Q to 9.2Q

Models real numbers as approximations. Scientific languages offer at least float and double. Most use IEEE 754.

  • Precision: Accuracy of the fractional part
  • Range: Min/max representable value
⚠ Important

Floating point values are approximations. 0.1 + 0.2 ≠ 0.3 in most languages due to binary representation.

Stores a fixed number of decimal digits using Binary Coded Decimal (BCD). Used in COBOL; C# provides a decimal type.

✓ Advantage
  • Exact decimal arithmetic — essential for financial computations
✗ Disadvantages
  • Wastes memory (BCD is inefficient)
  • Limited range

Boolean: Two-element range (true/false). Could be a single bit but addressing limitations mean bytes are usually used. Improves readability over using ints.

Character: Stored as numeric codings:

  • ASCII: 7-bit, 128 characters
  • Unicode UCS-2: 16-bit, most natural languages
  • Unicode UCS-4 / UTF-32: 32-bit, used in Java and C#

Array Types

Array
An aggregate of homogeneous data elements where individual elements are identified by their position relative to the first element. Indexing is a mapping: indices → elements.

Index Range Checking

LanguageRange Check?
C, C++, Perl, Fortran❌ No range checking — undefined behavior on out-of-bounds
Java, ML, C#✅ Runtime range checking — throws exception
 
// C++ — fixed heap-dynamic array
int* p = new int[size];   // size known at runtime
// ... use p ...
delete[] p;               // must manually free

// Java — all arrays are fixed heap-dynamic
int[] arr = new int[size]; // range checked at access

Rectangular vs Jagged Arrays

Rectangular Array
  • All rows same length
  • True 2D structure
  • Used in Fortran, Ada
  • C#, F# support via int[,]
Jagged (Ragged) Array
  • Rows can have varying lengths
  • Array of arrays
  • C, C++, Java use this model
  • More flexible but less cache-friendly

Some languages support operations on entire arrays rather than only element-by-element access. This is a major writability improvement over manually writing loops for every transformation.

Slices

A slice extracts part of an array as a unit. It is a referencing mechanism, not a distinct type. 

# Python slices
vector = [3, 6, 9, 12, 15, 18, 21]
vector[3:6]      # [12, 15, 18]
vector[0:7:2]    # [3, 9, 15, 21]

# Ruby slices
list.slice(2, 2)
list.slice(1..3)
Why Slices Matter

Slices make sub-array operations much more readable and are one of the biggest usability improvements over early fixed-array models.

Associative Arrays

An associative array is an unordered collection of key-value pairs where user-defined keys replace numeric subscripts.

LanguageNameNotes
PerlHashexists, delete, keys, values, each
PythonDictionaryMaps keys to object references
RubyHashKeys can be arbitrary objects
PHPUnified arrayOne structure behaves as both indexed and associative
LuaTableCan act as an array, associative array, or record
 
# Python dictionary
salaries = {"Gary": 75000, "Perry": 57000}
salaries["Perry"] = 58850

# Perl hash
%salaries = ("Gary" => 75000, "Perry" => 57000);
$salaries{"Perry"} = 58850;

Record Types

Record
A possibly heterogeneous aggregate of data elements (called fields), each identified by name.
-- Ada record example
type Employee_Record_Type is record
    First_Name : String(1..20);
    Last_Name  : String(1..20);
    Hourly_Rate: Float;
end record;

-- Field access via dot notation
Employee_Record.Hourly_Rate := 25.50;
📝 COBOL Elliptical References

COBOL allows elliptical references — you can omit record names if the field name is unambiguous. So FIRST, FIRST OF EMPLOYEE-NAME, and FIRST OF EMPLOYEE-RECORD can all be equivalent. Other languages require fully qualified references: record.subrecord.field.

Tuple Types

A tuple is similar to a record, except its components are not named. Tuples are widely used to return multiple values from a function.

LanguageTuple StyleKey Property
Python(3, 5.8, "apple")Immutable; indexed from 0
ML(3, 5.8, "apple")At least two elements; selectors like #1
F#let tup = (3, 5.8, "apple")Usually unpacked with pattern matching
 
# Python tuple unpacking
point = (4, 9)
x, y = point

// F# multiple assignment from a tuple
let a, b, c = tup

List Comprehensions

Python and F# support comprehension-style construction, which greatly improves writability by combining iteration, filtering, and construction into one expression. 

# Python list comprehension
[x ** 2 for x in range(12) if x % 3 == 0]   # [0, 9, 36, 81]

Union Types

Union
A type whose variables can store values of different types at different execution times. Only one value is held at a time; all members share the same memory location.
Free Unions (C, C++, Fortran)
  • No type checking support
  • Programmer must track current type
  • Accessing wrong field = garbage, no error
Discriminated Unions (ML, Haskell, F#)
  • Include a discriminant/tag field indicating current type
  • Accessing wrong field raises a runtime error
  • Type-safe
// C — free union (unsafe)
union flexType {
    int   intEl;
    float floatEl;
};
union flexType u1;
u1.intEl = 12;
float x = u1.floatEl;  // LEGAL — but reads garbage!

Pointer and Reference Types

Pointer
A variable that holds a memory address. The value nil/null/nullptr means it doesn't refer to any cell. Used for indirect addressing and/or dynamic heap management.

Pointer Operations

// C++ pointer operations
int val = 42;
int* p = &val;   // ASSIGNMENT: p holds address of val
int  j = *p;    // DEREFERENCING: j gets the value at p

The Two Pointer Problems

⚠ Dangling Pointers

A pointer to a deallocated heap variable. The memory has been freed, but the pointer still holds the old address. Accessing it causes undefined behavior — one of the most dangerous bugs in C/C++.

⚠ Memory Leakage (Lost Heap Variables)

An allocated heap variable with no pointer to it. The programmer allocated memory, then overwrote the pointer without freeing it. The memory can never be freed — the program gradually consumes more RAM. 

// Memory leak example
int* p = new int(5);  // allocate
p = new int(10);      // overwrite p — first allocation is now LEAKED

References

C++ references are like pointers but safer — they don't work with raw memory addresses and must be initialized on declaration.

// C++ reference
int v = 10;
int& ref = v;    // ref IS v — any change to ref changes v
ref = 20;         // v is now 20

// Used as pass-by-reference parameter
void increment(int& p) { p++; }
increment(v);     // v becomes 21
LanguagePointer/Reference Model
C, C++Full pointer arithmetic; void*; explicit new/delete
JavaReferences only (no raw pointers); GC handles deallocation; only for objects (not primitives)
C#References like Java, plus C++ style pointers in unsafe blocks

Type Checking

Type Checking
Ensuring that the operands of an operator are compatible types. A compatible type is either legal for the operator, or can be implicitly converted (coercion) to a legal type.
📝 Generalized Operators & Operands

For type-checking purposes, subprogram calls and assignments count too: f(1.2) — operator is f, operand is 1.2. a = 43 — operator is =, operands are a and 43.

Strong Typing

Strongly Typed Language
One in which all type errors are always detected. Note: "strength" is usually comparative — "Language A is more strongly typed than B."
LanguageStrengthReason
C, C++Not strongly typedParameter type checking can be avoided; unions are not type-checked
Java, C#Mostly strongly typedImplicit type errors are always detected; explicit casting can still cause errors (e.g., casting Person to Student)

Coercion and Reliability

Coercion — implicit type conversion — can weaken typing. When a compiler silently converts types, it may mask accidental errors. 

// Suppose b and d are both int, and x is float
float result = a + d;   // intended: a + b
                          // compiler coerces int d to float — no error!
                          // the typo goes undetected
🔑 Key Point

C++ has many coercions → less reliable. Java has about half of C++'s assignment coercions → more reliable. Ada has very few implicit coercions → most reliable.

Type Equivalence

Defines when values of two types can be used interchangeably without coercion.

Name Equivalence
  • Two variables are type-equivalent only if they have the same type name
  • Easy to implement
  • More restrictive — subprogram args must exactly match parameter types
Structure Equivalence
  • Two types are equivalent if they have identical structures
  • More flexible — two separately declared types with same structure are compatible
  • Harder to implement; can be too permissive

Chapter 6 Summary

  • Primitive types: Integer, float, complex, decimal, boolean, character. Design choices reflect hardware constraints and application needs.
  • Arrays: 5 categories of subscript/storage binding from static to heap-dynamic. Java uses fixed heap-dynamic; Python/JS use heap-dynamic. Slices and aggregate operations improve writability.
  • Associative arrays: Key-value collections used in Perl hashes, Python dictionaries, Ruby hashes, PHP arrays, and Lua tables.
  • Tuples and lists: Tuples package fixed multiple values; list models differ sharply between functional languages and Python-style dynamic lists.
  • Unions: Free unions (C) are unsafe; discriminated unions (ML/F#) include a tag that enables type checking.
  • Pointers: Enable heap access but introduce dangling pointer and memory leak problems. Java avoids this with garbage collection and references only.
  • Strong typing: Always detecting type errors. C/C++ are not; Java/C# are mostly. Coercion weakens type safety.
  • Type equivalence: Name equivalence is restrictive but simple; structure equivalence is flexible but harder to implement.
