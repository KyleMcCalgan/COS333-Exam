Prolog in Practice

Writing real Prolog — syntax, unification, recursion, list processing, the cut operator, and full worked examples for all three prac tasks.

🔑 Mental Model Shift

Stop thinking "how do I compute this?" and start thinking "what is true about this?" You are writing logical descriptions, not algorithms. Prolog's engine figures out the execution — your job is to define truth correctly.

Prolog Syntax — The Complete Picture

The Anatomy of a Rule

grandparent(X, Z) :- parent(X, Y), parent(Y, Z).
head / consequent neck — "if" body / antecedent (comma = AND)

Syntactic Rules — Memorise These

RuleCorrectWrong
Every clause ends with . fact(a). fact(a) ← missing period, silent error
Atoms start lowercase router1, alice Router1 ← this is a variable!
Variables start uppercase or _ X, MyVar, _Unused x, myVar ← these are atoms
Conjunctions use commas a(X), b(X) a(X) and b(X) ← not valid
Head is a single term foo(X) :- bar(X). foo(X), baz(X) :- bar(X). ← multiple heads invalid
Facts need no body computer(pc1). computer(pc1) :- true. ← works but redundant
Empty list is [] process([]). process([|]). ← invalid
is evaluates arithmetic Y is X + 1 Y = X + 1 ← this only unifies structurally

The Difference Between = and is

= — Unification
X = 3 + 2.
% X becomes the STRUCTURE 3+2
% NOT the number 5
is — Arithmetic Evaluation
X is 3 + 2.
% X becomes 5
% RHS must be fully instantiated
⚠ The is Rule

All variables on the right-hand side of is must already be instantiated before the is goal is evaluated. And the variable on the left must not yet be instantiated. You cannot do Sum is Sum + 1 — Prolog variables are not mutable.

Unification — How Prolog Matches

Unification is the core mechanism of Prolog. When you write a query or a rule body, Prolog tries to unify terms — finding variable bindings that make two terms identical.

Unification Rules

  • Two atoms unify iff they are identical: alice = alice → true, alice = bob → false.
  • A variable unifies with anything and gets bound to that value: X = alice → X = alice.
  • Two structures unify iff they have the same functor, same arity, and all corresponding arguments unify: f(a, X) = f(a, b) → X = b.
  • A variable unifies with a structure — X = f(a, b) → X = f(a, b).
  • Two different uninstantiated variables unify and become co-references.
?- foo(X, b) = foo(a, Y).
X = a, Y = b.        % both variables unified

?- [H | T] = [1, 2, 3].
H = 1, T = [2, 3].   % list unification

?- [H | T] = [].
false.               % empty list has no head

Instantiation During Resolution

When Prolog tries to satisfy a goal, it looks for a matching fact or rule head. Variables in the goal get instantiated to the values needed for the match. If later a subgoal fails, Prolog undoes these instantiations (backtracking) and tries the next candidate.

📝 Scope of Instantiation

Variables are local to a clause. Each time a rule is invoked, Prolog creates a fresh copy of all variables. So calling parent(X, Y) twice in the same rule body gives you two independent sets of fresh variables, not the same X and Y.

List Patterns — The Building Blocks

Every list predicate you write uses one or more of these canonical patterns. Internalise these and recursive list predicates become straightforward.

Empty list
process([]).
% matches [] only
Single element
process([X]).
% matches [anything]
% X bound to that element
Head + tail
process([H | T]).
% H = first element
% T = rest (always a list)
Two elements + tail
process([A, B | T]).
% A = 1st, B = 2nd
% T = rest (may be [])
Anonymous head
process([_ | T]).
% ignore the head value
% T = tail
Result in parameter
transform([H|T], [H2|R]) :-
    H2 is H * 2,
    transform(T, R).
% No "return" — result IS a param
⚠ The Tail is Always a List

In [H | T], T is always a list — possibly []. If the list is [a], then H = a and T = []. This is why the base case for empty list is almost always needed.

Writing Recursive Rules — The Formula

Almost all useful list predicates in Prolog follow the same structural template. Learn this template and apply it to every new predicate you write.

myPred([], BaseResult). % ← (1) base case: empty list

myPred([Head | Tail], Result) :- % ← (2) recursive case
    myPred(Tail, SubResult), % ← (3) recurse on tail
    combine(Head, SubResult, Result). % ← (4) build result from head + subresult

Step-by-step: How to Design a New Predicate

  1. Name and arity: What does success mean? What are all the parameters (inputs AND outputs)?
  2. Base case: What is the answer for the simplest input (empty list, single element)?
  3. Recursive case: If I already knew the answer for the tail, how would I compute the answer for [Head | Tail]?
  4. Order: Always put the base case clause first. Prolog matches top-to-bottom.
  5. Test with small examples: Trace manually before running.

Reading Traces — Step by Step

The tracing model has four events: Call Exit Fail Redo. Understanding traces is essential for debugging and for the exam.

Worked Trace: member(b, [a, b, c])

member(Element, [Element | _]).
member(Element, [_ | Lis]) :- member(Element, Lis).
Call: member(b, [a, b, c])
Try clause 1: member(b, [b | _])? head=a ≠ b → no match
Call: member(b, [a, b, c]) % try clause 2
Match: [_ | Lis] = [a, b, c] → Lis = [b, c]
Call: member(b, [b, c]) % recurse on tail
Try clause 1: member(b, [b | _])? head=b = b ✓
Exit: member(b, [b, c])
Exit: member(b, [a, b, c])
true.

Worked Trace with Backtracking: likes(jake, X), likes(darcie, X)

likes(jake, chocolate).
likes(jake, apricots).
likes(darcie, licorice).
likes(darcie, apricots).
Call: likes(jake, _X) % subgoal 1 — find what jake likes
Exit: likes(jake, chocolate) % X = chocolate (first fact)

Call: likes(darcie, chocolate) % subgoal 2 — test with X = chocolate
Fail: likes(darcie, chocolate) % darcie doesn't like chocolate

Redo: likes(jake, _X) % backtrack: try next value for X
Exit: likes(jake, apricots) % X = apricots (second fact)

Call: likes(darcie, apricots) % try subgoal 2 again
Exit: likes(darcie, apricots) % ✓ darcie does like apricots

X = apricots.
📝 Subgoal Order Efficiency

Notice: the query checked all of jake's likes for each attempt. If you put the more restrictive subgoal first, Prolog backtracks less. For example, if darcie only likes one thing, query likes(darcie, X), likes(jake, X) would be much more efficient.

The Cut Operator !

The cut (!) is a special goal that always succeeds, but prevents backtracking past the point it appears. It "commits" Prolog to the current clause choice. 

% Without cut — re-querying after the base case tries the recursive case:
getValues(_, [], []).
getValues(Min, [H|T], [H|R]) :- H > Min, getValues(Min, T, R).
getValues(Min, [H|T], R) :- H =< Min, getValues(Min, T, R).

% With cut — once base case matches, no backtracking into recursive cases:
getValues(_, [], []) :- !.
getValues(Min, [H|T], [H|R]) :- H > Min, !, getValues(Min, T, R).
getValues(Min, [H|T], R) :- getValues(Min, T, R).
✓ What Cut Does
  • Succeeds immediately as a goal
  • Prevents backtracking into the current clause's alternatives
  • Fixes the choice of clause for the current goal
  • Used to prevent redundant re-evaluation of base/recursive cases
✗ What Cut Does NOT Do
  • Does not make Prolog "stop" or "return"
  • Does not affect other goals outside the clause
  • Does not prevent trying other solutions to the overall query
  • Is not required — programs work without it; it's an optimisation
📝 Practical 3 — Cut is Optional

The spec notes you may get a spurious false on re-querying without a cut. This is not penalised. Adding a cut is an optimisation, not a correctness requirement. Focus on getting the logic right first.

Task 1 — Network Connectivity

This task models a network topology using only three types of facts, then builds rules for connectivity at increasing levels of abstraction. The key is understanding the bidirectionality constraint and building up from small predicates to large ones.

The Network Topology

The Facts (only these three types allowed)

computer(pc1). computer(pc2). computer(pc3).
computer(pc4). computer(pc5). computer(pc6).

router(router1). router(router2). router(router3).

connection(pc1,    router1).
connection(router1, pc2).
connection(router1, router2).
connection(router3, router2).
connection(router2, pc3).
connection(pc4,    router3).
connection(pc5,    pc6).
⚠ Only These Three Fact Types

The spec is strict: no other facts. You cannot add connected(router1, pc1). as a shortcut for bidirectionality. That must come from a rule.

Step 1 — Handle Bidirectionality

connection/2 facts are directional (A→B is a fact, B→A is implied). Create a helper predicate that treats both directions as equivalent: 

% connected/2: true if there is a connection in either direction
connected(X, Y) :- connection(X, Y).
connected(X, Y) :- connection(Y, X).
📝 Why a Helper?

Wrapping bidirectionality in its own predicate means every higher-level predicate only needs to call connected once, rather than repeating both directions. Keep each predicate focused on one job.

Step 2 — routersConnected/2

True when two routers X and Y are connected directly or through one intermediate router. The spec guarantees at most three routers, so at most one hop between them.

Predicate routersConnected(X, Y)

Strategy

  • Direct case: X and Y are both routers and directly connected.
  • One-hop case: there is an intermediate router Z such that X–Z is connected and Z–Y is connected.
  • Both cases must be bidirectional (handled by connected).

Implementation

routersConnected(X, Y) :-
    router(X), router(Y),
    connected(X, Y).

routersConnected(X, Y) :-
    router(X), router(Y),
    router(Z),
    connected(X, Z),
    connected(Z, Y).

Expected Queries

?-routersConnected(router1, router2).true
?-routersConnected(router3, router1).true
?-routersConnected(router1, pc1).false
📝 Why check router(X) and router(Y)?

Without these guards, variables in queries like routersConnected(X, Y) could bind to computers or other atoms. Always restrict to the expected types.

Step 3 — computerConnectedToRouter/2

Predicate computerConnectedToRouter(X, Y)

Strategy

X is a computer and Y is a router and they are directly connected (either direction).

computerConnectedToRouter(X, Y) :-
    computer(X), router(Y),
    connected(X, Y).

Expected Queries

?-computerConnectedToRouter(pc1, router1).true
?-computerConnectedToRouter(pc2, router1).true
?-computerConnectedToRouter(pc1, router2).false

Step 4 — computersConnected/2

Predicate computersConnected(X, Y)

Three Cases

  1. Direct: X and Y are computers directly connected (e.g., pc5–pc6).
  2. Same router: X and Y are both connected to the same router R.
  3. Via router chain: X is connected to router Rx, Y is connected to router Ry, and Rx and Ry are connected (directly or through a third router).
% Case 1: directly connected computers
computersConnected(X, Y) :-
    computer(X), computer(Y),
    connected(X, Y).

% Case 2: both connected to the same router
computersConnected(X, Y) :-
    computer(X), computer(Y),
    computerConnectedToRouter(X, R),
    computerConnectedToRouter(Y, R).

% Case 3: connected through (potentially different) routers
computersConnected(X, Y) :-
    computer(X), computer(Y),
    computerConnectedToRouter(X, Rx),
    computerConnectedToRouter(Y, Ry),
    routersConnected(Rx, Ry).

Expected Queries

?-computersConnected(pc5, pc6).true
?-computersConnected(pc1, pc4).true
?-computersConnected(pc4, pc1).true (bidirectional)
?-computersConnected(pc1, pc5).false
?-computersConnected(router1, pc1).false (router1 is not a computer)

Task 2 — getValuesGreaterThan/3

Filter a list, keeping only values strictly greater than a threshold. Preserve the original order. The result is the third parameter.

Thinking It Through

📥
Parameters

Min, input list, output list (result — values > Min)

🎯
Base Case

Empty input → empty output. Done.

🔀
Recursive Cases

Head > Min → include head. Head ≤ Min → skip head. Both recurse on Tail.

📤
Building Output

When including: output = [Head | SubResult]. When skipping: output = SubResult.

Predicate getValuesGreaterThan(Min, InputList, ResultList) 
% Base case: empty input → empty output
getValuesGreaterThan(_, [], []).

% Recursive case 1: head IS greater — include it
getValuesGreaterThan(Min, [H | T], [H | R]) :-
    H > Min,
    getValuesGreaterThan(Min, T, R).

% Recursive case 2: head is NOT greater — skip it
getValuesGreaterThan(Min, [H | T], R) :-
    H =< Min,
    getValuesGreaterThan(Min, T, R).

Manual Trace: getValuesGreaterThan(5, [-1, 6, 0, 8, -5], X)

Call: gVGT(5, [-1, 6, 0, 8, -5], X)
Head=-1, -1 =< 5 → skip. Recurse on [6, 0, 8, -5]

Call: gVGT(5, [6, 0, 8, -5], X1)
Head=6, 6 > 5 → include. Recurse on [0, 8, -5]

Call: gVGT(5, [0, 8, -5], X2)
Head=0, 0 =< 5 → skip. Recurse on [8, -5]

Call: gVGT(5, [8, -5], X3)
Head=8, 8 > 5 → include. Recurse on [-5]

Call: gVGT(5, [-5], X4)
Head=-5, -5 =< 5 → skip. Recurse on []

Call: gVGT(5, [], X5)
Exit: X5 = [] % base case

Unwind: X4=[], X3=[8], X2=[8], X1=[6,8], X=
 X = [6, 8].

Expected Queries

?-getValuesGreaterThan(5, [], X).X = [].
?-getValuesGreaterThan(5, [-1, 4], X).X = [].
?-getValuesGreaterThan(5, [-1, 6, 0, 8, -5], X).X = [6, 8].
📝 Adding the Cut (optional)

On re-querying, Prolog may try the "include" case using the "skip" clause and fail. You can add cuts to prevent this. Place ! after the comparison guard: 

getValuesGreaterThan(_, [], []) :- !.
getValuesGreaterThan(Min, [H|T], [H|R]) :-
    H > Min, !,
    getValuesGreaterThan(Min, T, R).
getValuesGreaterThan(Min, [_|T], R) :-
    getValuesGreaterThan(Min, T, R).

Note: with a cut, the third case no longer needs H =< Min because the cut ensures we only reach it if case 2 didn't match.

Task 3 — countConsecutivePairs/2

Count consecutive pairs of identical elements. A pair is consumed as a unit — so [a, a, a] yields 1 (first two form a pair, leaving one lone a). [a, a, a, a] yields 2 (pairs: [a,a] then [a,a]).

The Key Insight — Pairs Are Consumed

🔑 Pair semantics

After consuming a pair [X, X | Rest], you continue from Rest — not from [X | Rest]. This is what makes [a,a,a] = 1 pair, not 2. The third a is not the start of a new pair.

Thinking It Through

📥
Parameters

Input list, output count

🎯
Base Case

Empty list → 0 pairs. Also: single element → 0 pairs.

✅
Pair found case

List starts with [X, X | Rest] → count 1 + recurse on Rest

⏭
No pair case

Head ≠ second element → recurse on [Second | Rest]

Predicate countConsecutivePairs(List, Count) 
% Base case 1: empty list → 0 pairs
countConsecutivePairs([], 0).

% Base case 2: single element → 0 pairs
countConsecutivePairs([_], 0).

% Recursive case 1: found a pair (first two elements are equal)
% CONSUME BOTH and continue from Rest
countConsecutivePairs([X, X | Rest], Count) :-
    countConsecutivePairs(Rest, SubCount),
    Count is SubCount + 1.

% Recursive case 2: no pair (first two elements differ)
% Advance by ONE element (keep the second for potential future pairing)
countConsecutivePairs([_ | T], Count) :-
    countConsecutivePairs(T, Count).

Manual Trace: countConsecutivePairs([b, b, a, c, c, c, d], X)

Call: ccP([b, b, a, c, c, c, d], X)
[b, b | Rest]: b=b → PAIR found. Rest=[a,c,c,c,d]

Call: ccP([a, c, c, c, d], X1)
[a, c | ...]: a≠c → no pair. Advance to [c,c,c,d]

Call: ccP([c, c, c, d], X2)
[c, c | Rest]: c=c → PAIR found. Rest=[c,d]

Call: ccP([c, d], X3)
[c, d | ...]: c≠d → no pair. Advance to [d]

Call: ccP([d], X4)
Exit: X4 = 0 % base case: single element

Unwind: X3=0, X2=0+1=1, X1=1, X=1+1=2
X = 2.

Verify the [a, a, a] = 1 case

Call: ccP([a, a, a], X)
[a, a | Rest]: a=a → PAIR. Rest=[a]

Call: ccP([a], X1)
Exit: X1 = 0 % single element base case

Unwind: X = 0 + 1 = 1
X = 1. ✓ (not 2)

Expected Queries

?-countConsecutivePairs([], X).X = 0.
?-countConsecutivePairs([a, b], X).X = 0.
?-countConsecutivePairs([b, a, a, a, c], X).X = 1.
?-countConsecutivePairs([b, b, a, c, c, c, d], X).X = 2.
?-countConsecutivePairs([a, a, a, a], X).X = 2.
?-countConsecutivePairs([x, x, x], X).X = 1.

Common Mistakes & How to Fix Them

✗ Wrong
computer(PC1).
router(Router1).

PC1 is a variable, not the atom pc1. This matches anything.

✓ Correct
computer(pc1).
router(router1).

If you forget the empty list base case, every recursive predicate will eventually fail when it hits [].

✗ Wrong
% Only recursive case — will fail on []
myPred([H|T], R) :- myPred(T, R).
✓ Correct
myPred([], []).
myPred([H|T], R) :- myPred(T, R).
✗ Wrong
Count = SubCount + 1.
% Count = the STRUCTURE (SubCount+1)
% not the numeric value
✓ Correct
Count is SubCount + 1.
% Count = SubCount evaluated plus 1

The connection facts only go one direction. Without handling both directions, computersConnected(pc2, pc1) will fail even though computersConnected(pc1, pc2) succeeds. Always wrap directional facts in a bidirectional helper. 

% BAD: will fail for connection(router1, pc2) when called as connected(pc2, router1)
routerToComputer(R, C) :- connection(R, C).

% GOOD: bidirectional helper handles both orderings
connected(X, Y) :- connection(X, Y).
connected(X, Y) :- connection(Y, X).

Without type guards, queries like routersConnected(X, Y) will enumerate computers and other atoms as routers. Always add router(X) / computer(X) guards. 

% BAD: X and Y could be anything
routersConnected(X, Y) :- connected(X, Y).

% GOOD: guards restrict to routers
routersConnected(X, Y) :-
    router(X), router(Y),
    connected(X, Y).

For countConsecutivePairs, when a pair [X, X | Rest] is found, you must recurse on Rest — not on [X | Rest]. Recursing on [X | Rest] would try to pair the second element with the third again, double-counting.

✗ Wrong
countConsecutivePairs([X, X | R], C) :-
    countConsecutivePairs([X | R], Sub),
    C is Sub + 1.
% [a,a,a] would give 2 (wrong!)
✓ Correct
countConsecutivePairs([X, X | R], C) :-
    countConsecutivePairs(R, Sub),
    C is Sub + 1.
% [a,a,a] gives 1 (correct)

Prolog Practical — Quick Reference

  • Atoms lowercase, variables uppercase — never mix these up; it causes silent logic errors.
  • Every clause ends with . — missing periods cause syntax errors or silent failures.
  • = unifies, is evaluates arithmetic — use is for any computation.
  • List patterns: [] (empty), [H|T] (head+tail), [A, B|T] (two elements+tail), [_|T] (ignore head).
  • Recursive predicates: base case first, always. Recurse on the tail.
  • Bidirectionality in Task 1: wrap connection/2 in a connected/2 helper that tries both directions.
  • Type guards: add computer(X)/router(X) to prevent variables binding to wrong types.
  • Pair consumption in Task 3: after matching [X, X | Rest], recurse on Rest not [X | Rest].
  • Cut !: optional optimisation to suppress spurious false on re-query. Not required for correctness.
