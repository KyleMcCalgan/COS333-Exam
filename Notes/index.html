Key Themes Across All Chapters

Central Idea

Programming languages make design choices at every level — from how names are resolved to how types are checked to how methods are dispatched. Each choice involves trade-offs between writability, readability, reliability, and efficiency. The same four criteria from Chapter 1 resurface throughout every subsequent chapter.

The Static/Dynamic Axis

One of the most consistent themes across all chapters is the trade-off between static (compile-time) and dynamic (runtime) approaches:

  • Static: determined at compile time — more efficient, more reliable, less flexible.
  • Dynamic: determined at runtime — more flexible, higher overhead, harder to debug.
  • Most modern languages use a mix (statically typed but with dynamic dispatch, e.g. Java).

Implementation Questions

The exam has one Scheme (Chapter 15) and one Prolog (Chapter 16) implementation question, both involving list processing. Practice is essential — understanding the concepts is not enough, you must be able to write correct code under time pressure in the actual interpreter.

What to Expect — Past Paper Analysis

Based on four years of exam papers (2022–2025), the following patterns emerge. This is the most likely shape of the 2026 exam.

Exam Format (consistent every year)
  • 40 marks total, 3 hours.
  • 1 Scheme implementation question — always 5 marks.
  • 1 Prolog implementation question — always 5 marks.
  • ~28–31 MCQ questions for the remaining 30 marks (mix of 1-mark single-select and 2–3-mark multi-select with penalty for wrong selections).
  • Questions follow chapter order: Ch1/Ch2 early, then Ch5–Ch9, then Ch12, with implementations (Ch15/Ch16) at questions 4–7.

Implementation Questions (10 marks total)

Scheme — 5 marks

The Pattern

Every year: a simple numeric list processed with recursion using car, cdr, cons, and null?. The function gets progressively more nuanced each year — adding a third condition each time:

  • 2022: filter — keep positive odd values
  • 2023: map — double positive values
  • 2024: filter — remove positive odd values
  • 2025: filter+map — double positives, keep zeros, omit negatives

2026 prediction: Expect 2–3 conditions on a numeric list combining both filtering (omit/keep) and transformation (double, negate, halve). Practice writing cond-based recursive functions cold. Start with #lang racket. The base case is always the empty list returning '().

Prolog — 5 marks

The Pattern

Every year: a list proposition with arithmetic conditions. Uses multiple clauses with head unification or comparison operators + is:

  • 2022: remove all occurrences of X from list
  • 2023: double all elements that don't match X
  • 2024: double positive values, keep non-positives
  • 2025: keep negatives, negate positives, omit zeros — cut and not allowed

2026 prediction: Numeric list with 2–3 cases differentiated by sign or value. Use the multi-clause pattern (base case = empty list; recursive cases for each condition). Remember: Prolog arithmetic must use is; use X > 0 etc. as goals. No if-then-else, no built-in member/append.

MCQ Predictions by Chapter

Chapter 1 — Preliminaries ~3–4 marks · HIGH CERTAINTY

Guaranteed topics (appear every year):

  • Evaluation criteria trade-offs — e.g. "when readability improves, what suffers?" — every year without exception.
  • Orthogonality — explicitly stated "will definitely be tested". APL's operator precedence/associativity being orthogonal has appeared twice (2022, 2025).
  • Applying a criterion (readability, writability, reliability, cost) to a specific language feature or design decision in a real or hypothetical language.

2026 focus: Know the precise definitions of each evaluation criterion and be ready to explain a trade-off. Practice orthogonality with APL (single-rule precedence = orthogonal). Be able to apply criteria to a feature you've never seen before. Scenarios about portability and generality have appeared twice each.

Chapter 2 — Evolution of PLs ~1–2 marks · MEDIUM CERTAINTY
  • Comparing two languages on one evaluation criterion (e.g. "Pascal vs C — which is more readable?"). Every year has at least one.
  • Identifying a language's environment or main contribution from a description.
  • Questions about portability of specific languages: JavaScript scores high (2024), Java scores high (2025). Know why (e.g. JVM bytecode for Java, runs in any browser for JavaScript).

2026 focus: Drill the environment→effect→contribution triad for FORTRAN, ALGOL, COBOL, PL/I, SIMULA 67, Pascal, C, C++, Java. Know which languages were historically notable for portability, readability, or well-definedness.

Chapter 5 — Names, Bindings & Scopes ~2–3 marks · HIGH CERTAINTY
  • Lifetime categories (static, stack-dynamic, explicit heap-dynamic, implicit heap-dynamic) — asked in 2023. Know each definition precisely.
  • Binding times — asked in 2024. Know language design time, implementation time, compile time, load time, runtime.
  • Named constants (manifest vs static — 2023). Know what happens when a named constant is bound to a value at compile time vs runtime.
  • Special words — keywords vs reserved words (2025). Know the distinction: a keyword's meaning depends on context; a reserved word always means the same thing.
  • Static vs dynamic scoping — this feeds into the Ch9 binding trace questions below.

2026 focus: Scope and lifetime distinctions are reliably tested. Be ready for a code snippet asking you to classify a variable's lifetime, or to determine a variable's scope and lifetime separately (2022 Q6).

Chapter 6 — Data Types ~2–3 marks · MEDIUM CERTAINTY
  • Strong/weak typing — C++ is weakly typed due to unions and the way pointers work (2023). Know which features weaken a language's type system.
  • Unions and discriminated unions — appeared in 2024. Know: free unions = weak typing; discriminated unions add a tag field making them safer.
  • String types — static, limited dynamic, dynamic length strings and their trade-offs (2024).
  • List comprehensions in Python — asked in 2023 and 2025. Answer: they improve writability (concise syntax to build lists).
  • Pointers and reference types — implicit dereferencing, void pointers, dangling pointers.

2026 focus: Discriminated unions (tag field, reliability benefit), type equivalence (name vs structural), and the trade-offs of different string representations are reliable targets.

Chapter 7 — Expressions & Assignments ~4–5 marks · VERY HIGH CERTAINTY

The single most reliably tested topic in the whole exam:

  • Short-circuit evaluation + side effects — appeared in every single year (2022 Q12, 2023 not present, 2024 Q13, 2025 Q14). Always: given code with a logical and/or and a side effect (e.g. ++index, assignment in condition), which form causes a problem? The side effect not occurring when short-circuit skips the operand is the issue when the program relies on that side effect.
  • APL operator precedence = orthogonality — 2022 Q8 and 2025 Q11. In APL there is one rule: right-to-left with no precedence levels. This is more orthogonal but less aligned with mathematical conventions.
  • Conditional targets — appeared 3 years in a row (2023, 2024, 2025). They improve writability.
  • Coercions in Ada — Ada has no coercions, which improves reliability (2022 Q10, 2024 Q12).
  • Mixed-mode arithmetic comparisons (F# vs C++, Ada vs C) — appeared 2023, 2024, 2025. F# has no coercions → more reliable but less writable than C++.
  • Multiple-assignment targets in Ruby — improves writability (2025 Q16).

2026 focus: Know the short-circuit side-effect scenario cold — you will see it. Know whether each assignment/expression feature improves readability, writability, reliability, or cost, and be able to argue why.

Chapter 8 — Control Structures ~4–5 marks · HIGH CERTAINTY
  • Ruby blocks / yield trace — 2023 Q19 and 2025 Q21. A method uses yield in a loop; you trace what gets printed. Pattern: follow the loop iteration, each yield passes the current value to the block which prints it. Practice these traces.
  • Guarded commands trace — 2022 Q16 (if...fi) and 2024 Q17 (do...od). 2023 and 2025 had no guarded command trace. The alternation suggests 2026 is due for one — likely the iterative do...od form. Key rule: if no guard is true in a do, the loop terminates; if no guard is true in an if, it is a runtime error.
  • Nested selector disambiguation — appeared in 2023, 2024, 2025. Every year compares Python, Ruby, and/or Perl. Python uses indentation (reliable, readable, but opinionated); Perl uses if without special delimiters; Ruby uses end keyword. Know which is more writable, readable, reliable.
  • Counter-controlled loops — compared across languages every year. C is flexible but less reliable; Ada is more restricted and more reliable; Python's for-each is more reliable than C's for-loop.
  • User-located loop control — Java has labelled break (can exit named outer loop); C++ does not → Java is more writable here (2025 Q20).

2026 focus: Practise Ruby block traces (follow the yield values through the block). Expect a guarded-commands trace. Know the nested-selector comparison table (Python/Ruby/Perl/Java).

Chapter 9 — Subprograms ~5–6 marks · VERY HIGH CERTAINTY
  • Coroutine output trace — appeared in 2022 Q21, 2023 Q24, 2024 Q23. Absent in 2025. This strongly suggests 2026 will have one. Pattern: resume suspends the calling routine and transfers to the target. Trace which print runs before each resume and where control returns after the resumed routine hits its next resume. Each coroutine resumes from where it left off.
  • Ad-hoc / shallow / deep binding traces — appeared every year. A subprogram passed as a parameter: which environment is used when it's called?
    • Deep binding: referencing environment at the point the parameter was passed.
    • Shallow binding: referencing environment at the point the parameter is called.
    • Ad-hoc binding: with static scoping — uses the environment of the subprogram's definition site.
  • Parameter passing modes — every year. Know: pass-by-value (copy in), pass-by-result (copy out), pass-by-value-result (copy in+out), pass-by-reference (access path), pass-by-name (textual substitution). 2024 Q20 traced pass-by-name — the actual parameter expression is re-evaluated each access.
  • Closures in JavaScript — appeared in 2025 Q25. A function returning an inner function that captures outer variables. Know: thing = doSomething(10, 20) creates a closure that captures a=10, b=20; calling thing(5) gives 25.
  • Generics comparison C++/Java/C# — appeared every year:
    • C++: template instantiation — one version per unique type argument, resolved at compile time.
    • Java: type erasure — one version at compile time using Object; the compiler inserts casts.
    • C#: reification — one version per value type at runtime; one shared version for all reference types.
  • Default parameters / keyword parameters — 2022 Q18, 2023 Q20, 2025 Q22. Keyword parameters allow any parameter to be omitted regardless of position; positional parameters do not.

2026 focus: Coroutine trace is overdue (missed 2025). Practise binding traces with static and dynamic scoping together. Know parameter passing modes well enough to trace code. Generics C++ vs Java vs C# is a perennial question.

Chapter 12 — OOP Support ~4–5 marks · VERY HIGH CERTAINTY
  • Multiple inheritance diamond problem (implicit name collision) — appeared in 2022 Q26, 2023 Q28, and 2025 Q28 — practically guaranteed. The exact same question structure: A→B, A→C, D inherits B and C. An implicit collision happens when a method is defined only in A (inherited into both B and C, so D gets two copies). An explicit collision requires the same name in both B and C.
  • Java generics — version count — 2022 Q24 (written, 3 marks), 2023 Q27 (1 mark). Java type erasure means one version exists at compile time regardless of how many type arguments are used.
  • Subclasses vs subtypes — 2024 Q26, 2025 Q31. In Java a subclass is always a subtype (Liskov substitution holds). In C++ a subclass need not be a subtype (private derivation). Smalltalk: subclasses are always subtypes. Know the principle.
  • Dynamic vs static message binding — C++ lets you choose (virtual vs non-virtual methods); Java and Smalltalk use dynamic binding exclusively → C++ is more efficient but less orthogonal / reliable.
  • Ruby OOP — dynamic classes (can add methods at runtime), everything is an object, single inheritance only, modules for mixins. Dynamic classes reduce reliability (2022 Q23).
  • Smalltalk — pure OOP, everything is an object (even integers), highly orthogonal but very inefficient (all binding is dynamic, even primitives are objects). Appeared 2022, 2024, 2025.
  • C++ ADT representation hiding — friend functions and the requirement for class data members to appear in header files both detract from hiding (2024 Q24).
  • Java hiding representation — compiles to bytecode (.class file); only the class file is distributed. Javadoc is the public documentation, not the header (no header files). 2022 Q22, 2023 Q26.

2026 focus: Multiple inheritance diamond problem is near-certain. Know the differences between Smalltalk, C++, Java, C#, and Ruby on: exclusivity of objects, dynamic vs static binding, subclasses/subtypes, and how each hides ADT representations. Expect 1–2 multi-select questions worth 2 marks each with a penalty.
