Reasons for Studying Programming Languages

⚠ Exam Pattern

The exam will give you a specific scenario (e.g., a junior programmer, a manager, a language designer) and ask for one or more reasons relevant to that person. You cannot simply list any reason — your answer must logically connect to the given scenario. Study each reason with its real-world application in mind.

Understanding programming language concepts benefits anyone who works with software. Six key reasons, each with a distinct audience:

1. Better use of available languages
Knowing language features deeply allows you to write better programs using features you may have overlooked or misused. A programmer who understands closures will write cleaner code in JavaScript; one who understands type systems will avoid unsafe casts in Java.
2. Better ability to learn new languages
Languages share concepts (scoping rules, type systems, parameter passing). Once you understand these at a conceptual level, picking up a new language is much faster. The mechanism differs, but the idea is familiar.
3. Better understanding of the significance of implementation
Understanding how language features are implemented (e.g., dynamic dispatch, garbage collection, closures) helps you reason about performance, memory use, and correctness. A manager can make informed decisions about language choice for a project.
4. Better ability to design new languages
Studying existing languages — their successes and failures — is essential for anyone designing a DSL or a new general-purpose language. You avoid repeating historical mistakes.
5. Better understanding of the importance of implementation
Language features have costs. Understanding these helps explain why some features (e.g., dynamic typing, exceptions) are avoided in embedded or real-time systems.
6. Overall advancement of computing
A broader understanding of programming paradigms enables innovation. New paradigms (functional, logic, concurrent) expand what is computable or expressible.

Language Evaluation Criteria

Four primary criteria are used to evaluate and compare programming languages. Note that these criteria often conflict — improving one can harm another.

Central Framework

These four criteria — readability, writability, reliability, cost — are the lens through which every language feature in this course is evaluated. Be able to apply them to features in both real and hypothetical languages.

Readability

How easily can a program be read and understood? This is arguably the most important criterion since programs are read far more often than they are written.

Overall Simplicity

A language with a small number of basic constructs is easier to learn and read. Complications arise from:

  • Feature multiplicity — multiple ways to do the same thing (e.g., i = i + 1, i += 1, i++ in C). The reader must understand all forms.
  • Operator overloading — the same symbol means different things in different contexts, which can confuse readers if used inconsistently.

Orthogonality

⚠ High-Frequency Exam Topic

Orthogonality is guaranteed to appear on the exam. Understand the definition, the examples, and how it creates a trade-off between readability and writability.

Orthogonality
A small set of primitive constructs can be combined in a relatively unrestricted way, and every combination is legal and meaningful. The meaning of a combination is independent of the context in which it is used.

The term comes from mathematics: orthogonal vectors are independent — they do not affect each other. In a language, orthogonal features do not interact in special, context-dependent ways.

Example — IBM mainframe architecture

IBM mainframe assembly had two types of operands (registers vs memory) and three instruction types (register-register, register-memory, memory-memory). A fully orthogonal design would allow all 6 combinations. IBM's design only supported some, so programmers had to remember which combinations were forbidden — low orthogonality.

Example — VAX architecture vs IBM

The VAX was highly orthogonal: any operand mode (register, literal, indexed, indirect, etc.) could be used with any instruction. This makes a small set of rules describe a large language.

Orthogonality in high-level languages

  • ALGOL 68: extremely orthogonal. Any expression can appear anywhere an expression is expected. Very consistent but complex to implement and sometimes difficult to read.
  • C: moderately orthogonal but has exceptions (e.g., void functions, struct assignments were historically inconsistent).
  • Java: less orthogonal — primitive types are not objects, so they cannot be used wherever objects are expected. You cannot put an int in a generic collection directly.

Too much orthogonality is also a problem

High orthogonality means many combinations are legal — some of which are nonsensical or produce subtle errors. The compiler catches fewer mistakes because almost anything is valid. Readability suffers when legal expressions have surprising semantics.

Advantages of high orthogonality
  • Few special cases or exceptions to remember
  • Greater expressive power from small feature set
  • More predictable, consistent behavior
Disadvantages of high orthogonality
  • Many legal but meaningless combinations possible
  • Fewer compile-time error checks
  • Can reduce readability (too many valid ways to express an idea)

Data Types

Adequate data types improve readability. A language with a Boolean type lets you write flag = true instead of flag = 1, which is self-documenting.

Syntax Design

  • Identifier forms: meaningful, multi-word identifiers (e.g., totalBalance) improve readability.
  • Special words: reserved words like if, while, for with consistent syntax aid understanding.
  • Form and meaning: syntax should reflect semantics. Inconsistency (e.g., same keyword used for different meanings in different contexts) harms readability.

Writability

How easily can a language be used to create programs for a given domain?

Simplicity and Orthogonality

A smaller, orthogonal set of features is easier to learn and combine. However, having too few features forces verbose workarounds. There is an optimal range.

Support for Abstraction

Abstraction means defining and using structures that allow programmers to ignore detail. Two types:

  • Process abstraction — subprograms/functions. Instead of copying logic, you name it and call it.
  • Data abstraction — abstract data types (classes). Bundle data with its operations and hide the implementation.

Expressivity

Powerful, concise operations improve writability. Example: the for loop over a collection in Python is more expressive than manual index management in C. The increment operator ++ in C is more expressive but also adds to the special-case knowledge needed.

Reliability

A program is reliable if it performs its specified behaviour under all conditions.

Type Checking

Detecting type errors. The earlier errors are caught, the better. Static type checking (at compile time) is more reliable than dynamic (at runtime), which can allow programs to run with undetected type errors.

Exception Handling

The ability to intercept unexpected conditions at runtime and recover gracefully. Languages with good exception handling (Java, Ada) are more reliable than those without (early C).

Aliasing

Having two or more distinct names for the same memory location. This reduces reliability because modifying one alias unexpectedly changes the other. Pointer-heavy languages (C) are less reliable partly because of aliasing.

Readability and Writability

Both indirectly affect reliability: readable programs are easier to verify and maintain; writable programs with complex constructs may be harder to write correctly.

Cost

The total cost of a language includes many factors beyond just execution speed:

  • Training cost — how long it takes to train programmers to use the language.
  • Cost of writing programs — development time, writability.
  • Cost of compiling programs — compile time, tool requirements.
  • Cost of executing programs — runtime efficiency. (Often in conflict with reliability features like array bounds checking.)
  • Cost of language implementation — building compilers/interpreters.
  • Cost of poor reliability — bugs found late, in production, are far more expensive to fix.
  • Cost of maintenance — readability directly drives long-term maintenance cost.

Influences on Language Design

Computer Architecture

The dominant influence on imperative languages is the von Neumann architecture: a CPU, memory, and a connection (bus) between them. Programs store data in memory and fetch/execute instructions sequentially. This naturally maps to:

  • Variables as named memory locations
  • Assignment statements as the primary computation step (fetch, modify, store)
  • Iteration (loops) rather than recursion as the natural repetition mechanism

Functional and logic languages emerged partly as a reaction against von Neumann thinking.

Programming Methodology

Software engineering movements directly influenced language design:

  • 1950s–60s: focus on machine efficiency → FORTRAN, COBOL
  • 1960s–70s: structured programming (Dijkstra, Hoare) → Pascal, Algol
  • 1970s–80s: data abstraction → Ada, Modula-2
  • 1980s–90s: object-oriented programming → Smalltalk, C++, Java

Many modern languages are multi-paradigm: Python, Scala, and JavaScript support both OOP and functional styles.

Language Design Trade-offs

Language features often conflict with each other. Designers must make explicit trade-offs:

Example — array bounds checking: Checking array indices at runtime prevents buffer overflows and memory corruption, greatly improving reliability. However, it adds overhead to every array access, which can be significant in tight loops. C omits bounds checking for performance; Java and Python include it for reliability.

A hypothetical language that performs bounds checking only for the first access to an array and skips it thereafter would improve performance at the cost of reliability.

Example — APL operators: APL is extremely writable: a complete matrix manipulation can fit on one line using special operators. But those same programs are nearly unreadable to anyone who hasn't memorised the operator set. Writability was maximised at the cost of readability.

Example — operator overloading: Overloaded operators make programs shorter and more natural to write, but they can confuse a reader if the overloaded meaning is not obvious from context.

Example — pointers in C: Pointer arithmetic gives programmers tremendous flexibility (direct memory access, efficient data structures). But unrestricted pointer use is a major source of errors: dangling pointers, buffer overflows, memory leaks. Java removed pointer arithmetic entirely; Ada restricts it heavily. Safety improved, but some low-level flexibility was lost.

A uniform, orthogonal language is easier to learn but may lack special-purpose features that would make specific tasks much easier. Domain-specific languages sacrifice generality for expressivity in a narrow area.

Implementation Methods

There are three main methods for implementing a programming language. You need to understand the concepts, not the detailed internal steps.

Compilation
The source program is translated into machine code (or an intermediate form) by a compiler before execution. The resulting binary runs directly on hardware. High execution speed; errors detected at compile time. Examples: C, C++, FORTRAN, Ada.
Pure Interpretation
The source program is executed directly by an interpreter, statement by statement, with no prior translation. No separate compilation step; easier to debug interactively; much slower at runtime due to overhead of decoding each statement on every execution. Examples: early BASIC, some scripting languages. Rare today for full programs.
Hybrid Implementation
Source is translated to an intermediate language (e.g., bytecode), which is then interpreted or compiled at runtime. Combines portability (the intermediate form runs on any platform with an interpreter) with better performance than pure interpretation. Examples: Java (JVM bytecode), Python (.pyc bytecode), .NET (CIL).
Just-In-Time (JIT) Compilation

A further refinement of hybrid implementation: the bytecode interpreter compiles hot code paths to native machine code at runtime. This allows Java and JavaScript engines to approach the speed of compiled code while retaining portability.

Chapter 1 Summary

  • Readability: influenced by simplicity, orthogonality, data types, syntax. Most important long-term criterion.
  • Orthogonality: small primitive set + unrestricted combination. High orthogonality means few exceptions but can reduce safety.
  • Writability: how easily programs can be created for a domain. Requires simplicity + abstraction + expressivity.
  • Reliability: type checking, exception handling, reducing aliasing.
  • Cost: includes training, writing, compiling, executing, and maintaining programs.
  • Trade-offs: reliability vs cost, writability vs readability, flexibility vs safety.
  • Implementation: compilation (fast, static), interpretation (flexible, slow), hybrid (portable, moderate speed).
